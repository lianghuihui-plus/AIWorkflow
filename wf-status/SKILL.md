# 工作空间状态与健康检查

对话输出当前项目进度、待决策项、健康检查和下一步建议。不生成文件。

> 工具型 skill。在 workspace 目录下执行。

## 执行流程

### 步骤 0：运行内置校验器

优先运行当前 `wf-status` skill 目录下的校验器：

```text
python <wf-status-skill-dir>/tools/validate.py <workspace> --json
```

校验器只做确定性结构扫描，不修改文件。`wf-status` 以校验器结果作为健康检查事实基础，再补充解释、汇总和下一步建议。

如果校验器不可用，才回退到本文件后续的手工扫描规则，并在输出中说明校验器缺失。

### 步骤 1：读取文件

读 `CONTEXT.md`。不存在 → 报错：**「未找到 CONTEXT.md，当前目录不是 AIWorkFlow 工作空间」**，终止。

读 `ISSUES.md`。不存在 → 待决策数显示为 `未知（ISSUES.md 缺失，未扫描）`，跳过问题编号检查，并在健康检查中标记为 ⚠️。

读 `REVISIONS.md`。不存在 → 待处理修订数显示为 `未知（REVISIONS.md 缺失，未扫描）`，跳过修订编号检查，并在健康检查中标记为 ⚠️。

`wf-status` 是只读检查工具。缺少 `ISSUES.md` 或 `REVISIONS.md` 时可以降级展示为未知或 ⚠️；但执行 `wf` 的写入事务时，这些文件缺失会被门禁视为阻塞。

检查以下路径是否存在：

- `JOURNAL.md`
- `CHANGELOG.md`
- `AGENT.md`
- `REVISIONS.md`
- `prd/`
- `output/`

从 CONTEXT.md 的 `## 当前状态` 提取当前阶段和下一步；不要使用文件开头引用块中的阶段描述。从 `## 项目约束` 提取平台和代码仓库。
从 CONTEXT.md 的 `## 当前状态` 提取 `待处理产物`；如果校验器报告待处理产物与真实审核状态不一致，在健康检查中标记为 ❌ 或 ⚠️，并建议执行 `wf` 进入 `fix-workspace` 后重建状态快照。
如 `ISSUES.md` 存在，从中提取各阶段待决策问题数（按 `## 分析阶段` 等标题分组，统计 Q-xxx 条目）；如不存在，各阶段待决策数显示为 `未知`，不要输出“共 0 项”。
如 `REVISIONS.md` 存在，从 `## 待处理` 中统计状态为 `待处理` 或 `阻塞` 的 R-XXX 条目；如不存在，待处理修订数显示为 `未知`。

### 步骤 2：执行健康检查

健康检查只做确定性的一致性检查，不判断需求质量、方案优劣或代码正确性。

解析 `CONTEXT.md` 时以 `wf/contracts/context.md` 的字段和解析规则为准；如本文件中的复述规则与契约冲突，以契约为准。

能由校验器确定的问题，以校验器结果为准，避免 Agent 为状态检查读取全量 `output/` 产物。Agent 只需要读取校验器指出的异常文件和当前状态输出所需的摘要信息。

#### 检查项

| 检查项 | 规则 |
|-------|------|
| 工作空间文件 | 必要文件和目录存在；`CONTEXT.md` 缺失直接终止 |
| PRD 文件 | `prd/` 存在，且包含至少一个支持格式文件；否则标记 ⚠️ |
| 阶段产物 | 按当前阶段和下一步检查关键输入是否存在 |
| 审核状态 | 按 `contracts/review-status.md` 检查关键产物是否已确认 |
| 任务索引 | 按 `CONTEXT.md` 章节解析任务，检查重复、规格/报告/测试报告断链 |
| 待处理产物 | `CONTEXT.md` 待处理产物列表与阶段产物审核状态一致 |
| 代码仓库 | 当下一步是代码实现或测试生成时，`代码仓库` 必须非空、不能为 `无`，且路径可访问 |
| 问题编号 | 按 `ISSUES.md` 阶段 section 解析 Q-XXX，检查重复和阶段归属 |
| 修订编号 | 按 `REVISIONS.md` 解析 R-XXX，检查重复和待处理修订 |

#### 阶段产物规则

从 `CONTEXT.md` 的 `下一步` 字段提取下一动作。识别运行时能力名称和终态动作：`analyze-requirements`、`design-solution`、`generate-specs`、`implement-code`、`generate-tests`、`review-artifact`、`status`、`resolve-decision`、`fix-workspace`。

匹配时按“字段中包含名称”判断，例如 `下一步：implement-code` 视为代码实现。

如果下一步是 `status`：

- 当前阶段是 `tests_done`：阶段产物检查标记 ✅，说明“流程完成”。
- 当前阶段不是 `tests_done`：阶段产物检查标记 ⚠️，说明“下一步为 status，但阶段不是 tests_done”，建议修正 `CONTEXT.md`。

如果下一步是 `resolve-decision` 或 `fix-workspace`，阶段产物检查标记 ⚠️，说明当前处于阻塞恢复流程，建议先处理待决策或修复工作空间。

如果下一步是 `review-artifact`，阶段产物检查标记 ⚠️，说明当前存在待审核或需修改产物；建议用户确认通过、提出修订，或执行 `wf` 查看待审核清单。

如果存在待审核或需修改产物，但 `CONTEXT.md` 的 `待处理产物` 缺失或不一致，标记为 ❌ 或 ⚠️，建议执行 `wf` 进入 `fix-workspace` 并通过 `rebuild_context.py` 重建。

如果下一步字段为空或不包含任何已知动作：

- 当前阶段是 `tests_done`：阶段产物检查标记 ✅，说明“流程完成，无下一步”。
- 其他阶段：阶段产物检查标记 ⚠️，说明“下一步未识别”，建议修正 `CONTEXT.md`。

如果识别出以下能力，则按对应规则检查：

- 下一步是 `analyze-requirements`：`prd/` 应存在，且至少包含一个 `.md`、`.txt` 或 `.pdf` 文件。
- 下一步是 `design-solution`：`output/analysis.md` 应存在，且审核状态应为 `已确认`。
- 下一步是 `generate-specs`：`output/design.md` 和 `output/analysis.md` 应存在，且审核状态均应为 `已确认`。
- 下一步是 `implement-code`：`代码仓库` 必须非空、不能为 `无` 且路径可访问；`output/design.md` 中每个任务都应有对应 `output/specs/T-XXX.md`，且全部规格应已确认。
- 下一步是 `generate-tests`：`代码仓库` 必须非空、不能为 `无` 且路径可访问；存在至少一个“已完成但未测试”的任务，且对应 `output/reports/T-XXX.md` 应存在并已确认。
- 下一步是 `review-artifact`：至少存在一个阶段产物审核状态为 `待审核` 或 `需修改`。

`PRD 文件` 缺失支持格式时始终作为工作空间完整性 ⚠️；只有下一步是 `analyze-requirements` 时，才视为阶段产物输入问题。

#### 任务索引规则

按 `CONTEXT.md` 的 section 分别解析任务，不要在全文范围内直接统计 `T-XXX`，避免把同一任务在不同 section 中的正常出现误判为重复。

规格索引只支持标准模板位置：`## 当前状态` 下的 `- 规格：`。如果标准位置没有任务，但在 `CONTEXT.md` 其它位置发现具体任务号（匹配 `T-[0-9]{3}`，不包含 `T-XXX` 占位文本），标记 ⚠️“规格索引位置非标准”，不要静默当作无任务。

| 来源 | 解析范围 | 用途 |
|------|----------|------|
| 规格索引 | `## 当前状态` 下 `- 规格：` 后续缩进两空格的 `- T-XXX` 行；遇到非缩进字段行（如 `- 平台：`）或下一个 `##` 时结束 | 任务定义、重复任务、规格文件检查 |
| 代码产出 | `## 代码产出` 表格 | 已完成任务和报告文件检查 |
| 测试记录 | `## 测试记录` 表格 | 已测试任务和测试报告文件检查 |

检查规则：

- `规格索引` 中的 `暂无` 不算任务。
- `规格索引` 中包含 `T-XXX`，且该行不含 `✅ 已实现`、`暂停`、`跳过`、`待决策`、`阻塞` 的任务，视为未完成任务。
- `规格索引` 中包含 `T-XXX` 且该行含 `✅ 已实现` 的任务，不算待代码生成任务。
- `规格索引` 中包含 `✅ 已确认` 只能表示规格已确认，不算任务已实现。
- `规格索引` 中同一任务编号重复 → ❌。
- `规格索引` 中存在 `T-XXX` 但 `output/specs/T-XXX.md` 缺失 → ⚠️。
- `output/design.md` 中存在任务但缺少对应 `output/specs/T-XXX.md` → ⚠️；当下一步是 `implement-code` 时 → ❌。
- `output/design.md` 中存在任务但对应规格审核状态不是 `已确认` → ⚠️；当下一步是 `implement-code` 时 → ❌。
- `规格索引` 中没有任何 `T-XXX`，但下一步是 `implement-code` → ❌，建议先执行 `generate-specs` 或修正 `CONTEXT.md`。
- `规格索引` 中不存在未完成任务，但下一步仍是 `implement-code` → ❌，建议更新下一步或选择实际待开发任务。
- `代码产出` 中 `T-XXX` 状态为 `✅ 已完成` 但 `output/reports/T-XXX.md` 缺失 → ⚠️。
- `测试记录` 中 `T-XXX` 状态为 `✅ 已完成`，但 `output/test-reports/T-XXX.md` 缺失 → ⚠️。
- `规格索引` 中 `T-XXX` 标记为 `✅ 已测试` 时，只有该行同时包含对应任务号的 `output/test-reports/<任务号>.md`（如 `T-001` 对应 `output/test-reports/T-001.md`）才视为规格索引测试状态自洽；否则依赖 `## 测试记录` 表判断测试完成状态。
- 判断是否有待测试任务时，使用 `代码产出` 中已完成任务减去 `测试记录` 中已完成任务；集合非空才建议执行 `generate-tests`。
- 没有已完成但未测试任务，但下一步仍是 `generate-tests` → ⚠️；如果规格索引中至少存在一个 `T-XXX`，且全部任务均已标记 `✅ 已测试`，建议将阶段更新为 `tests_done`，下一步更新为 `status`。

#### 审核状态规则

按 `contracts/review-status.md` 解析阶段产物的 `## 审核状态`。

审核状态扫描范围必须包含：

- `output/analysis.md`
- `output/design.md`
- `output/specs/*.md`
- `output/reports/T-*.md`
- `output/test-reports/T-*.md`

- 产物缺少 `## 审核状态` → ⚠️；如果该产物是下一步必要输入 → ❌。
- 产物审核状态为 `待审核` → ⚠️，建议用户审核确认或提出修订；如果该产物是下一步必要输入 → ❌。
- 产物审核状态为 `需修改` → ⚠️，建议执行 `wf` 收敛修订；如果该产物是下一步必要输入 → ❌。
- 产物审核状态为 `已确认` → ✅。
- `tests_done` 状态下，所有 `output/test-reports/T-XXX.md` 都应为 `已确认`。
- `下一步=review-artifact` 但没有待审核或需修改产物 → ❌，建议修正 `CONTEXT.md`。

#### 问题编号规则

如果 `ISSUES.md` 不存在，跳过本检查项，仅在健康检查说明中标记文件缺失。

如果 `ISSUES.md` 存在，按 markdown 二级标题区间解析 Q 条目：从某个 `## 阶段标题` 开始，到下一个 `## ` 标题之前，均归属该阶段。

- 同一问题编号重复 → ⚠️。
- Q 条目不在 `## 分析阶段`、`## 设计阶段`、`## 实现阶段`、`## 测试阶段` 任一标题下 → ⚠️。
- 除明确包含 `状态：已解决` 的 Q 条目外，`ISSUES.md` 中存在任何 Q 条目即视为待决策问题 → ⚠️，但不默认阻塞下一步。
- 明确包含 `状态：已解决` 的 Q 条目不计入待决策数，但标记 ⚠️，建议归档到 `CHANGELOG.md` 并从 `ISSUES.md` 删除。

#### 修订编号规则

如果 `REVISIONS.md` 不存在，跳过本检查项，仅在健康检查说明中标记文件缺失。

如果 `REVISIONS.md` 存在：

- 同一修订编号重复 → ⚠️。
- `## 待处理` 中存在 `状态：待处理` 的 R 条目 → ⚠️，说明需要执行 `wf` 收敛修订。
- `## 待处理` 中存在 `状态：阻塞` 的 R 条目 → ⚠️，说明需要补充修订信息或处理阻塞原因。
- `## 已处理` 中缺少 `处理结果`、`更新产物` 或 `处理时间` 的 R 条目 → ⚠️。

#### 状态分级

| 状态 | 含义 |
|------|------|
| ✅ | 未发现结构问题 |
| ⚠️ | 发现不一致或待决策项，可能需要人工处理 |
| ❌ | 缺少下一步必需输入，或当前状态与下一步不一致，继续执行大概率失败 |

### 步骤 3：输出状态

```
# 工作空间状态

项目：XXX | 平台：XXX | 扫描时间：YYYY-MM-DD HH:MM

## 进度

当前阶段：XXX
下一步：XXX

## 待决策

如果 `ISSUES.md` 存在，使用以下模板：

| 阶段 | 待决策数 |
|------|---------|
| 分析阶段 | X |
| 设计阶段 | X |
| 实现阶段 | X |
| 测试阶段 | X |

> 共 X 项待人工决策，详见 ISSUES.md

如果 `ISSUES.md` 缺失，使用以下模板：

> 待决策数量未知（ISSUES.md 缺失，未扫描）

## 待处理修订

如果 `REVISIONS.md` 存在，输出：

> 共 X 项待处理修订，Y 项阻塞修订，详见 REVISIONS.md

如果 `REVISIONS.md` 缺失，输出：

> 待处理修订数量未知（REVISIONS.md 缺失，未扫描）

## 待审核产物

如果存在待审核或需修改产物，输出：

> 共 X 个待审核产物，Y 个需修改产物。请确认通过或提出修订。

同时列出 `CONTEXT.md` 中的待处理产物；如果校验器指出列表不一致，说明需要先重建 `CONTEXT.md` 状态快照。

如果不存在，输出：

> 当前没有待审核产物。

## 健康检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 工作空间文件 | ✅ | 必要文件完整 |
| PRD 文件 | ✅ | prd/ 中存在 X 个支持格式文件 |
| 代码仓库 | ✅ | 代码仓库路径已配置 |
| 阶段产物 | ❌ | 下一步是 implement-code，但规格索引中没有未完成任务 |
| 审核状态 | ❌ | output/specs/T-001.md 尚未确认 |
| 任务索引 | ⚠️ | T-003 缺少 output/specs/T-003.md |
| 问题编号 | ⚠️ | 实现阶段存在 2 个待决策项 |
| 修订编号 | ⚠️ | 存在 1 个待处理修订 |
```

> 表格内容必须按实际检查结果替换，不得照抄示例状态。

### 步骤 4：输出建议

简短建议（不超过两句话）：

建议按以下优先级输出。主建议只输出最高优先级；详细检查模式列出全部问题。

1. 存在 ❌：指出最关键的阻塞项；若是缺输入则建议补齐，若是状态不一致则建议修正 `CONTEXT.md`。
2. 存在非待决策的 ⚠️ 一致性问题：指出最影响下一步的问题，建议先修正工作空间状态。
3. 有待决策项：指出哪个阶段待决策最多，建议优先处理。
4. 有待处理修订：建议执行 `wf` 收敛修订。
5. 有待审核产物：建议用户审核确认或提出修订。
6. 无未决策项、无待处理修订、无待审核产物且无一致性问题：按工作流顺序推荐下一个未完成的阶段。

### 详细检查模式

用户明确说「详细检查」「严格检查」「健康检查」「validate」时，在上述输出后追加详细问题列表：

有详细问题时使用以下模板：

```
## 详细问题

| 严重级别 | 类型 | 问题 | 建议 |
|----------|------|------|------|
| ⚠️ | 任务索引 | CONTEXT.md 中 T-003 缺少 output/specs/T-003.md | 修正 CONTEXT.md 后执行 wf |
```

没有详细问题时使用以下模板：

```
## 详细问题

未发现详细一致性问题。
```
