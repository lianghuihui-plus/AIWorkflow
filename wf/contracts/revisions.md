# 产物契约：REVISIONS.md

## 路径

`REVISIONS.md`

## 职责

记录用户主动提出的产物修订意见。它用于把对话中的修改意图持久化，供 `wf` 后续收敛产物、同步下游并审计来源。

`REVISIONS.md` 不用于记录需求/设计本身，也不替代 `ISSUES.md`：

- `ISSUES.md`：Agent 发现不确定项，需要用户做决策。
- `REVISIONS.md`：用户主动提出明确修改意见，需要 Agent 收敛产物。

## 必需章节

```markdown
# 用户修订

## 待处理

暂无

## 已处理

暂无
```

## 修订条目

```markdown
### R-001 — {修订摘要}

- **目标产物：** output/analysis.md
- **修订类型：** 需求纳入调整 / 设计调整 / 规格调整 / 测试调整 / 其他
- **用户意见：** {用户原始或整理后的修订意见}
- **影响范围：** analysis / design / specs / code / tests
- **状态：** 待处理
```

处理后移动到 `## 已处理`，并补充：

```markdown
- **处理结果：** {已完成的收敛动作}
- **更新产物：** {产物路径列表}
- **处理时间：** YYYY-MM-DD HH:MM
```

无法处理时保留在 `## 待处理`，状态改为 `阻塞`，并补充：

```markdown
- **阻塞原因：** {原因}
- **下一步：** {需要用户补充的内容或建议执行的动作}
```

## 编号规则

- 修订编号格式为 `R-[0-9]{3}`。
- 新修订编号从所有章节中已有最大编号递增。
- 重复编号无效。
- `## 待处理` 和 `## 已处理` 内的修订条目必须分别按 `R-XXX` 升序排列。

## 写入规则

- 用户通过对话提出明确修订时，Agent 必须先写入 `REVISIONS.md`，再进行收敛；新条目写入 `## 待处理` 中按编号升序的正确位置。
- 用户直接修改产物中的审核区时，可以不新增修订条目；但如果影响下游产物，收敛时应在 `JOURNAL.md` 记录来源。
- `wf` 只自动处理 `## 待处理` 中状态为 `待处理` 的修订；状态为 `阻塞` 的修订必须先按其阻塞原因完成决策、补输入或修复状态。
- 修订收敛完成后，不删除条目，必须移动到 `## 已处理` 归档，并按 `R-XXX` 升序插入，不得直接插到章节顶部。
- 任一章节存在修订条目时，必须删除该章节的 `暂无` 占位；章节没有条目时才保留单独的 `暂无`。
- 修订收敛成功后，目标产物审核状态必须回到 `待审核`，等待用户重新确认。
- 修订阻塞时，目标产物审核状态保持 `需修改`，`修订来源` 指向阻塞的 R-XXX。
- 修订阻塞如果需要人工决策，必须同步写入 `ISSUES.md` 并进入 `blocked_by_decision`。
- 修订阻塞如果源于缺输入或状态不一致，必须进入对应的 `blocked_by_missing_input` 或 `blocked_by_inconsistent_state`。
- 修订处理后的运行时日志按 `runtime.md` 处理。
