# 产物契约：output/analysis.md

## 路径

`output/analysis.md`

## 必需章节

- `# 需求分析 — {项目名称}`
- `## 审核状态`
- `## PRD 来源`
- `## 文件清单`
- `## 需求概要`
- `## 功能需求`

## 功能需求结构

`## 审核状态` 必须符合 `contracts/review-status.md`。新生成或修订后的 `output/analysis.md` 状态必须为 `待审核`；只有用户确认后才能改为 `已确认` 并进入技术设计。

`## 需求概要` 是 `CONTEXT.md` 中需求概要的权威来源。需求分析能力只维护 `output/analysis.md` 中的需求概要；`CONTEXT.md` 的 `## 需求概要` 由运行时或 `rebuild_context.py` 从 `output/analysis.md` 同步生成。

`## 功能需求` 必须先包含需求纳入决策表，再包含需求详情。

### 需求纳入决策表

```markdown
### 需求纳入决策表

| ID | 标题 | 所属模块 | 来源 | 处理方式 |
|---|---|---|---|---|
| REQ-001 | {标题} | {模块} | PRD-01 |  |
```

`处理方式` 是人工审核入口，只能由用户填写。Agent 初次生成 `需求纳入决策表` 时必须留空，不得替用户预填 `纳入`、`暂不纳入` 或 `待决策`。

用户填写时允许值：

- `纳入`
- `暂不纳入`
- `待决策`

其他列由 AI 维护，用户通常不需要修改。

### 需求详情

每条候选需求都必须有详情块。详情块不得因为 `处理方式` 变为 `暂不纳入` 或 `待决策` 而删除。

```markdown
### 需求详情

#### REQ-001 — {标题}

**所属模块：** {模块}

**变更内容：**
- {变更}

**处理逻辑：**
- {逻辑}

**约束条件：**
- {约束}

**需求来源：** PRD-01

**不纳入原因：** 

**原文引用：**
> {从 PRD 中摘录的原文，保持原意，不改写}
```

`处理方式=暂不纳入` 时，必须在对应需求详情块中填写 `不纳入原因`。`处理方式=纳入` 或空值时，`不纳入原因` 可以留空。

## 用户修订收敛规则

`需求纳入决策表` 是人工审核和修订的优先入口。Agent 只负责生成候选需求和来源依据，不决定纳入或暂不纳入。用户修改 `处理方式` 后，后续 `wf` 执行需求分析或修订需求时，必须先读取表格，并只以 `处理方式` 作为人工决策依据。

收敛规则：

- `处理方式` 为空：该需求处于待人工选择状态，不得进入技术设计。
- `处理方式=纳入`：该需求详情必须保留；`不纳入原因` 可以留空。
- `处理方式=暂不纳入`：该需求详情必须保留，并在详情块中填写 `不纳入原因`；不得删除或移动该需求详情。
- `处理方式=待决策`：该需求不得进入技术设计，必须写入 `ISSUES.md` 的分析阶段问题。
- 表格与详情冲突时，以表格中的 `处理方式` 为准。
- 收敛完成后，必须追加 `JOURNAL.md`，说明已根据需求纳入决策表同步 `output/analysis.md`。

## 编号规则

- PRD 编号使用 `PRD-01`、`PRD-02`，按文件名顺序分配。
- 需求编号使用 `REQ-001`、`REQ-002`，所有候选需求共用一套编号。

## 质量规则

- 每份 PRD 文件必须出现在文件清单中。
- 必须包含 `## 需求概要`，概括 PRD 范围、候选需求数量和关键约束。
- 审核状态必须存在，且进入技术设计前必须为 `已确认`。
- 空 PRD 文件必须标注。
- 每条需求必须出现在 `需求纳入决策表` 中。
- 初次生成需求分析时，`需求纳入决策表` 中每条需求的 `处理方式` 必须为空，等待用户人工填写。
- 用户确认 `output/analysis.md` 前，每条需求的 `处理方式` 必须已填写为 `纳入`、`暂不纳入` 或 `待决策`；存在空值时不得确认。
- 每条需求详情必须具备所属模块、变更内容、处理逻辑、约束条件、来源和原文引用。
- `处理方式=暂不纳入` 的需求必须在详情块中具备不纳入原因。
- 平台归属不清或字段缺失时，必须写入分析阶段问题。
