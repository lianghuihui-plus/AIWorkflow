# 能力：生成开发规格

## 适用意图

当技术方案已存在，工作空间需要生成逐任务、逐文件、逐修改点的实现规格时使用。

## 前置条件

- `output/design.md` 存在。
- `output/analysis.md` 存在。

## 输入

- `output/design.md`：任务总览、技术目标、依赖、模块架构、接口/方法定义、数据结构、设计约束和影响范围。
- `output/analysis.md`：用于生成关键行为的需求行为和约束。
- 可选代码仓库：用于确认已有文件路径。

## 输出

- 符合 `contracts/spec.md` 的 `output/specs/T-XXX.md` 文件。
- 审核状态：`待审核`。
- 完成事件：`output/design.md` 中每个任务都有对应规格文件，且全部规格经用户确认后，由运行时产生 `specs_generated`。
- 条件触发必写：命中“不确定项”规则时，必须写入 `ISSUES.md` 的实现阶段问题。

## 规则

- `output/design.md` 中每个任务都要生成一个规格文件。
- 规格必须按任务读取 `技术目标`、`模块架构`、`接口/方法定义`、`数据结构`、`设计约束` 和 `影响范围`，不得只依据任务标题或需求摘要生成。
- `任务概述` 必须来自技术方案中的 `技术目标` 和关联需求，不复述 PRD 原文。
- `关键行为` 必须合并需求分析的行为要求和技术方案中的 `设计约束`。
- `实现方案` 必须承接技术方案中的模块架构、接口/方法定义和数据结构。
- `修改点` 必须落到受影响文件、接口、数据结构或测试/回归范围；`影响范围` 用于约束修改边界。
- 每个修改点必须包含文件、类型、位置（适用时）、内容和约束。
- 既有文件路径必须在代码仓库中搜索确认。
- 新文件必须标注 `[NEW]`。
- 关键行为必须保留来自需求分析和技术设计的行为约束。

## 不确定项

以下情况写入实现阶段问题：

- 无法确认既有文件路径。
- 技术方案指向不存在的模块，且没有明确的新文件位置。
- 需求行为与设计实现方式冲突。
- `AGENT.md` 或项目约定中缺少必要编码约束。

## 完成标准

- 每个任务都有对应规格文件。
- 每份规格都列出依赖和修改点。
- 运行时可以据此更新 `CONTEXT.md` 中完整任务索引。
- 每份生成或修订的规格审核状态为 `待审核`，等待用户确认后才能进入代码实现。
