# 能力：需求分析

## 适用意图

当工作空间需要分析 PRD、用户要求分析需求，或状态机下一步是 `analyze-requirements` 时使用。

## 前置条件

- `CONTEXT.md` 存在。
- `prd/` 存在，且至少包含一个 `.md`、`.txt` 或 `.pdf` 文件。

## 输入

- `CONTEXT.md`：项目名称、目标平台、可选代码仓库路径。
- `prd/`：所有 PRD 来源文件，按文件名字母序扫描。
- 可选 `output/analysis.md`：已有分析结果，用于保持需求理解稳定。

## 输出

- 符合 `contracts/analysis.md` 的 `output/analysis.md`。
- 审核状态：`待审核`。
- 完成事件：用户确认 `output/analysis.md` 后，由运行时产生 `analysis_completed`。
- 可选：写入 `ISSUES.md` 的分析阶段问题。

## 规则

- PRD 编号使用 `PRD-01`、`PRD-02`，按文件名顺序分配。
- 需求编号使用 `REQ-001`、`REQ-002`，所有候选需求共用一套编号。
- 先完整提取 PRD 内容，再按目标平台分流。
- 必须生成 `需求纳入决策表`，作为用户审核入口。
- Agent 只生成候选需求、所属模块、来源和原文引用；不得替用户填写 `处理方式`。
- 初次生成 `需求纳入决策表` 时，所有 `处理方式` 必须留空，等待用户人工选择。
- 用户只需要修改 `需求纳入决策表` 中的 `处理方式`；AI 维护其他列和详情内容。
- 修订已有 `output/analysis.md` 时，必须先读取 `需求纳入决策表`，并按 `处理方式` 收敛需求详情中的状态相关内容。
- 需求详情一旦生成必须保留；不得因为 `处理方式=暂不纳入` 或 `处理方式=待决策` 删除、移动该需求详情。
- `处理方式=暂不纳入` 时，必须在对应需求详情中记录不纳入原因。
- 每条需求必须保留可对照的 PRD 原文引用；原文引用应摘录来源段落或关键句，保持原意，不改写。
- 空文件必须在文件清单中标注。
- 如存在代码仓库路径，只用项目结构辅助判断平台和模块归属。

## 不确定项

以下情况写入分析阶段问题：

- 无法判断需求是否属于目标平台。
- 纳入需求缺少必要字段。
- 多份 PRD 存在冲突。
- 需求描述会改变行为，但预期行为不明确。

## 完成标准

- 每份 PRD 文件都出现在文件清单中。
- 每条需求都有来源 PRD 编号。
- 每条需求都出现在 `需求纳入决策表` 中。
- 初次生成时，`需求纳入决策表` 中所有 `处理方式` 均为空，等待用户人工填写。
- 用户确认 `output/analysis.md` 前，每条需求的 `处理方式` 必须已填写；存在空值时不得确认或进入技术设计。
- 每条需求都具备所属模块、变更内容、处理逻辑、约束条件、来源和原文引用，并保留在需求详情中。
- `处理方式=暂不纳入` 的需求在详情中具备不纳入原因。
- `处理方式=待决策` 的需求已写入 `ISSUES.md`，不得进入技术设计。
- `output/analysis.md` 的审核状态为 `待审核`，等待用户确认后才能进入技术设计。
