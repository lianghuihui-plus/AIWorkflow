# 能力：技术设计

## 适用意图

当需求分析已完成，工作空间需要任务拆解和技术方案设计时使用。

## 前置条件

- `output/analysis.md` 存在。

## 输入

- `output/analysis.md`：已纳入需求和行为细节。
- 可选 `output/design.md`：已有设计，用于修订时保持任务结构稳定。

## 输出

- 符合 `contracts/design.md` 的 `output/design.md`。
- 审核状态：`待审核`。
- 完成事件：用户确认 `output/design.md` 后，由运行时产生 `design_completed`。
- 条件触发必写：命中“不确定项”规则时，必须写入 `ISSUES.md` 的设计阶段问题。

## 规则

- 按可独立实现、可独立验证的关注点拆分任务。
- 同一模块内不可独立验证的关联改动应合并。
- 依赖只表达硬依赖，不把软顺序写成依赖。
- 如存在代码仓库路径，提出文件、接口、数据结构前必须先理解真实模块结构。
- 任务详情必须以技术方案视角组织：先说明可验证的技术目标，再说明模块设计、变更落点、约束和影响；不得在任务内复述 PRD 原文。
- `技术目标` 只写本任务要达成的可验证技术结果，不写业务背景、需求原文或待确认内容。
- `模块架构` 只写本任务涉及的模块、组件或层，以及职责归属、调用关系、数据流、事件流或边界变化；不得混入接口签名、字段清单、异常兜底或兼容细节。
- `模块架构` 涉及多个模块、跨层调用、事件流或状态流时，可以补充任务局部 Mermaid 图；简单单模块任务写明不适用原因。
- 接口定义必须足够具体，能支撑规格生成：文件路径、签名、入参类型、出参类型。
- `接口/方法定义` 只列本次需求相关且新增、修改、签名变化或职责变化的接口/方法；未变化的既有接口、复用调用和上下游依赖不得放入该列表。
- `接口/方法定义` 只描述文件、签名、职责、入参和出参；分支、兼容、幂等、异常、兜底、去重、阈值、状态切换等行为规则应放入 `模块架构`、`数据结构` 或 `设计约束`。
- `数据结构` 只写本次新增、修改、删除的数据类、对象、字段、枚举、配置或 schema；每条至少包含归属位置、结构名、变更类型、字段/成员/结构名、类型或取值范围。
- `数据结构` 不使用表格强制定列，不写未变化字段；无变化时写“无数据结构变更”。
- `设计约束` 只写影响实现和验证的兼容、异常、兜底、幂等、去重、阈值、状态切换、平台限制等规则。
- `影响范围` 只写受影响路径、模块、调用方、测试或回归范围，不重复展开设计细节。
- 必须生成 `设计视图`，用少量高信息密度 Mermaid 图降低人工审核成本；图只辅助审核，不替代任务详情中的结构化字段。
- 图中的说明文字必须使用中文，包括节点说明、边说明、状态说明和流程判断说明；类型名、方法名、字段名、文件名、接口名、类名、枚举值等代码标识可以保持英文或项目原始命名。
- 任务数大于 1 时，必须提供全局 `任务依赖图`；单任务方案可写不适用原因。
- 默认应提供全局 `类关系图`。存在代码仓库且方案涉及类、接口、组件、数据结构或模块对象时，必须用 `classDiagram` 展示核心关系；纯配置、纯文案、纯样式、纯资源替换、无代码仓库或无法确定类/组件关系时，必须写明不适用原因。
- 涉及跨模块调用、接口交互、事件、异步回调或外部服务时，必须提供模块交互图或时序图。
- 涉及状态切换、阈值、分支判断、幂等、去重或重置规则时，必须提供状态图或决策流程图。
- 不得机械地为每个任务都画图；任务局部图只在单个任务内部逻辑复杂时补充。

## 不确定项

以下情况写入设计阶段问题：

- 无法确定模块归属。
- 接口形态或数据模型需要产品/后端决策。
- 现有代码结构与需求描述矛盾。
- 任务拆分或合并需要用户判断。

## 完成标准

- 每条纳入需求至少映射到一个任务。
- `output/design.md` 包含 `设计视图`，并按触发条件提供 Mermaid 图或明确不适用原因。
- 每个任务包含技术目标、关联需求、依赖、模块架构、接口/方法、数据结构、设计约束和影响范围。
- 设计约束体现需求中的超时、校验、取消、兼容性或平台限制等要求。
- `output/design.md` 的审核状态为 `待审核`，等待用户确认后才能进入规格生成。
