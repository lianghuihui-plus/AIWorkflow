# 能力：代码实现

## 适用意图

当任务规格已存在，用户要求实现、继续开发或实现下一个待开发任务时使用。

## 前置条件

- 代码仓库路径非空、不是 `无`、可访问。
- `output/design.md` 中每个任务都有对应 `output/specs/T-XXX.md`，且所有目标规格审核状态均为 `已确认`。
- 目标 `output/specs/T-XXX.md` 存在，且审核状态为 `已确认`。
- 任务依赖已完成；如果用户明确指定未满足依赖的任务，必须先提示风险。

## 输入

- `output/specs/T-XXX.md`：任务概述、依赖、关键行为、实现方案、修改点。
- 代码仓库中的相关源码文件。

## 输出

- 限定在任务范围内的代码仓库改动。
- 符合 `contracts/code-report.md` 的 `output/report-T-XXX.md`。
- 审核状态：`待审核`。
- 完成事件：用户确认 `output/report-T-XXX.md` 后，由运行时产生 `task_implemented` 或 `all_tasks_implemented`。
- 可选：写入 `ISSUES.md` 的实现阶段问题。

## 规则

- 以实现任务目标为准，不盲从与真实代码矛盾的规格。
- 不超出当前任务相关文件和行为做额外优化。
- 报告中必须记录每个修改点的处理结果。
- 即使偏离是合理的，也必须写入报告。
- 需要人工决策的偏离必须写入 `ISSUES.md`。
- 不得为了掩盖实现偏离而修改 `output/specs/T-XXX.md`。

## 不确定项

以下情况写入实现阶段问题：

- 规格引用了不存在的文件、类、方法或不兼容架构。
- 规格实现方式无法满足关键行为。
- 存在更好的实现方案，但会改变设计意图或对外行为。
- 关键行为与修改点指令冲突。

## 完成标准

- 任务代码已实现，或有阻塞问题说明为什么停止。
- `output/report-T-XXX.md` 已生成。
- 报告列出所有修改文件和偏离说明。
- `output/report-T-XXX.md` 的审核状态为 `待审核`，等待用户确认后才能标记任务完成或进入测试生成。
