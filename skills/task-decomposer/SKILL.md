# 任务拆解

从 `requirements.md` 中读取已确认的需求，拆解为粗粒度的开发任务块并排序，产出 `tasks.md`。

> 本阶段只做**任务拆分和依赖排序**。不涉及技术方案、不推测文件路径、不写实现细节。这些留给 tech-designer 和 task-spec-generator。

## 配置读取

### 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

定位到 `workflow.yaml` 后，立即解析并执行：

**除工作流目录定位外，本阶段所有输入从 `workflow.yaml` 读取，不交互询问。**

### 读取并校验

1. 解析 `workflow.yaml`，校验 YAML 格式
2. 提取 `output.base_dir`（默认 `output`），记为 `OUT`
3. 提取 `project.name` — 用于文档标题

YAML 格式非法导致无法解析 → 终止并告知用户具体错误。后续步骤直接使用上述变量，不再重复读取。

### 配置字段说明

| 配置路径 | 含义 | 必填 |
|---------|------|------|
| `output.base_dir` | 输出根目录 | 是（默认 `output`） |
| `project.name` | 项目名称 | 否 |

## 工作流程

输出写入 `{OUT}/tasks.md`，结构见下方模板。

### 前置检查

检查 `{OUT}/requirements.md`：
- 不存在 → 终止并告知用户：**「未找到 requirements.md，请先执行 prd-analyzer」**
- 存在但「状态」不为「已确认」→ 终止并告知用户：**「需求文档尚未确认（当前状态：xxx），请先完成需求审核」**

扫描 requirements.md 中 `纳入 = Y` 的需求，如无任何已确认需求 → 终止并告知用户。

读取 `{OUT}/requirements.md` 全文，后续步骤基于其内容拆解任务。

> 任何步骤遇到不可恢复的错误，终止并将错误信息告知用户，不静默跳过。

---

### 完整流程（初版生成）

#### Step 1：逐需求粗拆

对每条 `纳入 = Y` 的需求，按以下规则拆成任务块：

**拆解粒度参考：**
- 一个独立模块的实现 → 一个任务
- 一个页面的交互逻辑 → 一个任务
- 一个接口/数据通道的对接 → 一个任务
- 一段独立的业务规则 → 一个任务

**拆分信号（满足任一即拆）：**
- 涉及不同模块或页面
- 开发和验证可以独立进行
- 明显有先后依赖
- 同一需求内涉及多个独立关注点（如入口逻辑 / 按钮处理 / 埋点 / 冲突处理 / 第三方对接），代码改动不在同一处

**合并信号（满足任一可合并）：**
- 同一模块/页面的多个关联改动
- 共享同一段代码路径
- 互相不可独立验证

#### Step 2：依赖分析与排序

所有任务拆完后，分析依赖并排序：

**区分依赖类型：**
- **硬依赖**：B 需要 A 的代码产出才能工作（如接口定义、数据结构、核心逻辑必须先存在）→ 在依赖列写明 `T-xxx`
- **软依赖**：B 和 A 属于同一需求的后续步骤，代码无耦合 → **不设为依赖**，在摘要或标题中注明关联关系即可

**排序优先级：**
- 无依赖的排前面 → 硬依赖被依赖的次之 → 同优先级按需求编号顺序
- 互相不依赖的任务视为可并行

#### Step 3：扫描全局约束

检查每条已纳入需求的「约束条件」，如果存在以下信号词，标记为全局约束：
- `所有` / `全部` / `每个` / `统一` / `覆盖 XX`
- `任何` / `不论` / `无论何种`

对每个全局约束：
- **受影响子任务判定**：检查已拆解出的各子任务，判断其涉及范围是否与该全局约束相关（关键词匹配或语义相关）。如「覆盖当课所有编程任务」→ 所有涉及编程步骤入口、字段透传、开关判定的任务都受影响
- 在受影响任务的详情中以 `**⚠️ 全局约束：** 描述内容` 单行标出
- 若多个任务受同一全局约束影响，在任务标题后加 `[G]` 标记
- **无法明确判断时**：列入「待澄清」，不要擅自假设

#### Step 4：生成 tasks.md

按下方模板生成 `{OUT}/tasks.md`：

> **需求摘要的条目排序规则：核心判定逻辑放前面，边界和约束放后面。**
> 不要把关键逻辑和次要信息平铺在同一层级。

**摘要分类映射指引**（从需求的「变更内容」+「约束条件」中提取，不要求每条都凑齐所有分类，有就写，没有就省略）：

| 分类前缀 | 含义 | 来源 |
|---------|------|------|
| 核心逻辑 | 主要判定规则、核心业务流程 | 变更内容的主干 |
| 入口条件 | 触发条件、入口场景 | 变更内容中的入口/场景描述 |
| 校验规则 | 格式校验、边界条件 | 约束条件中可独立验证的部分 |
| 交互约束 | 超时、重试、字段口径、接口约束 | 约束条件中的技术限制 |
| 数据变更 | 字段增删改、透传规则 | 变更内容中的数据变化 |

**生成规则：**
- 任务 ID 按生成顺序分配 T-001、T-002…
- 依赖列：无依赖填 `—`；有硬依赖填 `T-xxx`
- 状态列初稿统一填「待开发」
- 需求摘要条目使用 `- 分类前缀：内容` 格式（按排序规则排列）
- 全局约束影响的任务在标题后加 `[G]` 标记，详情中加 `**⚠️ 全局约束：**` 单行

```markdown
# 任务分解 — [项目名称]

> 生成日期：YYYY-MM-DD
> 来源：requirements.md（已确认）
> 状态：待审核

### T-001 — 登录接口接入

**关联需求：** REQ-001

**需求摘要：**
- 核心逻辑：手机号+验证码登录接口对接
- 校验规则：手机号 11 位数字格式校验，错误立即提示不发送请求
- 交互约束：验证码 60 秒倒计时，超时自动过期；接口 3 秒超时，成功返回 token，失败返回错误码

**⚠️ 全局约束：** 登录态需覆盖应用内所有页面

**依赖：** 无

### T-002 — 登录页交互开发

**关联需求：** REQ-001

- 入口条件：用户点击登录入口进入登录页
- 交互约束：验证码发送后按钮置灰，倒计时结束恢复；登录中展示 loading

**⚠️ 全局约束：** 登录态需覆盖应用内所有页面

**依赖：** T-001

### T-003 — 头像上传实现

**关联需求：** REQ-002

**需求摘要：**

- 校验规则：图片大小不超过 5MB，格式仅支持 jpg/png
- 交互约束：上传进度条展示，失败支持重试（最多 3 次）

**依赖：** 无
```

### 修订模式

#### Step 1：处理审核

检查 `{OUT}/tasks.md` 的文档状态：
- 状态为「已确认」→ 告知用户：**「任务分解已完成（当前 vX.X），无需修订」**，终止
- 状态为「待审核」→ 继续处理
- 状态为其他值或无法解析 → 视为「待审核」，继续处理，并在操作提示中标注异常状态值

**待澄清条目：**
- 扫描每个 CL 详情块中的「人工决策」行
- 有内容 → AI 复核：
  - 已明确解决 → 按决策调整任务（合并/拆分/新增），CL 状态改「已解决」。**⚠️ 已解决的 CL 条目和详情块必须保留在文档中，不得删除。**
  - 不清晰或不完整 → 保持「待处理」，AI 复核附说明（告知用户哪些点仍需澄清）
- 人工决策为空 → 保持「待处理」

扫描用户消息中提到的 CL ID 或任务 ID，将对应审核意见应用到匹配条目；无明确 ID 时按关键词匹配。已解决条目不再处理，除非审核人明确要求。

#### Step 2：追加修订记录

在「修订记录」表格末尾追加一行（新条目始终追加在表格最后一行之后，不在顶部插入）。版本号读取现有记录最后一行递增，无记录时从 v1.0 开始。

#### Step 3：检查完成标准

所有 CL 条目的「状态」均为「已解决」→ 文档状态改为「已确认」。
任一项未满足 → 保持「待审核」。

> 修订完成后执行「输出前自检」。

## 输出前自检

生成或修订 `tasks.md` 后，确认以下全部通过：

- [ ] 每条 `纳入 = Y` 的需求都有对应的任务
- [ ] 所有任务都分配了 T 编号，且编号连续
- [ ] 依赖关系只表达硬依赖，软依赖未设为依赖
- [ ] 全局约束影响的任务已标记 `[G]` 或 `⚠️ 全局约束`
- [ ] 任务总览表中的每个任务在详情区都有对应小节
- [ ] 待澄清总览表与详情块一一对应，无孤立的条目
- [ ] 修订记录版本号连续，最新一行在最下
- [ ] 文档状态与实际完成标准一致
