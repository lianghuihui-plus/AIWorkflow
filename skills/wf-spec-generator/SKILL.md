# 开发规格生成

将技术方案转化为逐文件逐修改点的编码规格，产出 `output/specs/T-XXX.md`。

> 在 workspace 目录下执行。

## 前置检查

当前目录存在 `CONTEXT.md`。不存在 → 报错终止。
`output/design.md` 存在。不存在 → 报错：**「未找到 design.md，请先执行 wf-tech-designer」**，终止。

---

## 执行流程

### Step 1：读取输入

读 `CONTEXT.md`，提取平台、代码路径。
读 `output/design.md`，提取所有任务（T-xxx）及技术设计详情。
读 `output/analysis.md`，提取每条功能需求的变更内容、处理逻辑（用于 spec 的「关键行为」）。

### Step 2：生成规格文件

`output/specs/` 目录不存在则先创建。

对每个任务逐修改点生成规格。每条规格包含：

- **目标文件** — 精确到文件路径（需搜索代码仓库确认，新增文件标注 `[NEW]`）
- **修改类型** — 新增 / 修改 / 删除
- **修改位置** — 函数/类/结构体名称
- **修改内容** — 具体的编码指令
- **约束条件** — 来自 AGENT.md 的平台编码规范

**文件路径确认：** 必须搜索代码仓库确认文件存在。不确定则搜索，搜不到则写入 ISSUES.md。禁止使用「需确认」占位。

```markdown
# T-001 — [任务标题]

> 关联需求：REQ-001
> 平台：[平台名称]

---

## 任务概述

[2-3 句话说明这个任务要做什么]

## 依赖

| 依赖任务 | 说明 |
|---------|------|
| T-XXX | [依赖原因] |
| — | 无依赖 |

> 从 design.md 任务总览表提取

## 关键行为

[运行时行为描述，运行时行为描述，从 analysis.md 的「变更内容」+「处理逻辑」搬运]

- [正常路径的交互流程]
- [异常/边界处理]
- [状态变化]

## 实现方案

- **模块架构：** [涉及模块]
- **接口/方法：** [需要暴露的接口]
- **数据结构：** [涉及的数据结构]
- **约束条件：** [关键约束]

> 以上从 design.md 提取

---

## 修改点

### 修改点 1：[简述]

- **文件：** `path/to/file.ext` [NEW]
- **类型：** 新增
- **内容：** [具体的编码指令]
- **约束：** [来自 AGENT.md 的编码规范]

### 修改点 2：[简述]

- **文件：** `path/to/existing.ext`
- **类型：** 修改
- **位置：** [函数/类名]
- **内容：** [具体的编码指令]
- **约束：** [约束]
```

### Step 3：更新工作空间文件

**更新 CONTEXT.md：**

- `## 当前状态` → `阶段` 改为 `规格生成完成`，`下一步` 改为 `执行 wf-code-generator`
- `规格` 下用实际任务列表替换 `暂无`：

```
  - T-001 — [标题]（X 个修改点）
  - T-002 — [标题]（X 个修改点）
```

**写入 ISSUES.md（文件路径无法确认等不确定项）：**

在 `## 实现阶段` 下按以下格式逐条追加，编号按已有 Q-xxx 递增：

```
### Q-001 — [问题简述]

- **问题：** [具体描述]
- **AI 建议：** [建议方案]
- **影响：** [影响的文件或任务]
- **提出：** YYYY-MM-DD wf-spec-generator

---

**人工决策：**
**状态：** 待决策
```

**写入 JOURNAL.md：**

在文件末尾追加当天日期标题（如已存在则复用），然后追加：

```
### HH:MM — wf-spec-generator 完成

- 生成 X 个规格文件
- 发现问题 X 项，写入 ISSUES.md
```

## 输出前自检

- [ ] 每个任务都有对应的规格文件
- [ ] 文件路径经过搜索确认
- [ ] 每条规格包含完整的五个字段（文件、类型、位置/内容、约束）
- [ ] 有疑问的地方已写入 ISSUES.md
