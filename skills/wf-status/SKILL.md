# 工作空间状态与健康检查

对话输出当前项目进度、待决策项、健康检查和下一步建议。不生成文件。

> 工具型 skill。在 workspace 目录下执行。

## 执行流程

### Step 1：读取文件

读 `CONTEXT.md`。不存在 → 报错：**「未找到 CONTEXT.md，当前目录不是 AIWorkFlow 工作空间」**，终止。

读 `ISSUES.md`。不存在 → 待决策数显示为 `未知（ISSUES.md 缺失，未扫描）`，跳过问题编号检查，并在健康检查中标记为 ⚠️。

检查以下路径是否存在：

- `JOURNAL.md`
- `CHANGELOG.md`
- `AGENT.md`
- `prd/`
- `output/`

从 CONTEXT.md 提取：项目名称、当前阶段、下一步、平台。
如 `ISSUES.md` 存在，从中提取各阶段待决策问题数（按 `## 分析阶段` 等标题分组，统计 Q-xxx 条目）；如不存在，各阶段待决策数显示为 `未知`，不要输出“共 0 项”。

### Step 2：执行健康检查

健康检查只做确定性的一致性检查，不判断需求质量、方案优劣或代码正确性。

#### 检查项

| 检查项 | 规则 |
|-------|------|
| 工作空间文件 | 必要文件和目录存在；`CONTEXT.md` 缺失直接终止 |
| PRD 文件 | `prd/` 存在，且包含至少一个支持格式文件；否则标记 ⚠️ |
| 阶段产物 | 按当前阶段和下一步检查关键输入是否存在 |
| 任务索引 | 按 `CONTEXT.md` section 解析任务，检查重复、spec/report/test-report 断链 |
| 问题编号 | 按 `ISSUES.md` 阶段 section 解析 Q-XXX，检查重复和阶段归属 |

#### 阶段产物规则

从 `CONTEXT.md` 的 `下一步` 字段提取 skill 名称。匹配时按“字段中包含 skill 名称”判断，例如 `执行 wf-prd-analyzer` 视为下一步是 `wf-prd-analyzer`。

如果下一步字段为空或不包含任何已知 `wf-*` skill：

- 当前阶段是 `测试完成`：阶段产物检查标记 ✅，说明“流程完成，无下一步”。
- 其他阶段：阶段产物检查标记 ⚠️，说明“下一步未识别”，建议修正 `CONTEXT.md`。

如果识别出以下 skill，则按对应规则检查：

- 下一步是 `wf-prd-analyzer`：`prd/` 应存在，且至少包含一个 `.md`、`.txt` 或 `.pdf` 文件。
- 下一步是 `wf-tech-designer`：`output/analysis.md` 应存在。
- 下一步是 `wf-spec-generator`：`output/design.md` 应存在。
- 下一步是 `wf-code-generator`：`规格索引` 中应存在至少一个未完成任务，且该任务对应的 `output/specs/T-XXX.md` 应存在。
- 下一步是 `wf-test-generator`：存在至少一个“已完成但未测试”的任务。

`PRD 文件` 缺失支持格式时始终作为工作空间完整性 ⚠️；只有下一步是 `wf-prd-analyzer` 时，才视为阶段产物输入问题。

#### 任务索引规则

按 `CONTEXT.md` 的 section 分别解析任务，不要在全文范围内直接统计 `T-XXX`，避免把同一任务在不同 section 中的正常出现误判为重复。

规格索引只支持标准模板位置：`## 当前状态` 下的 `- 规格：`。如果标准位置没有任务，但在 `CONTEXT.md` 其它位置发现具体任务号（匹配 `T-[0-9]{3}`，不包含 `T-XXX` 占位文本），标记 ⚠️“规格索引位置非标准”，不要静默当作无任务。

| 来源 | 解析范围 | 用途 |
|------|----------|------|
| 规格索引 | `## 当前状态` 下 `- 规格：` 后续缩进两空格的 `- T-XXX` 行；遇到非缩进字段行（如 `- 平台：`）或下一个 `##` 时结束 | 任务定义、重复任务、spec 文件检查 |
| 代码产出 | `## 代码产出` 表格 | 已完成任务和 report 文件检查 |
| 测试记录 | `## 测试记录` 表格 | 已测试任务和 test-report 文件检查 |

检查规则：

- `规格索引` 中的 `暂无` 不算任务。
- `规格索引` 中包含 `T-XXX`，且该行不含 `✅`、`暂停`、`跳过`、`待决策`、`阻塞` 的任务，视为未完成任务。
- `规格索引` 中包含 `T-XXX` 且该行含 `✅` 的任务，不算待代码生成任务。
- `规格索引` 中同一任务编号重复 → ⚠️。
- `规格索引` 中存在 `T-XXX` 但 `output/specs/T-XXX.md` 缺失 → ⚠️。
- `规格索引` 中没有任何 `T-XXX`，但下一步是 `wf-code-generator` → ❌，建议先执行 `wf-spec-generator` 或修正 `CONTEXT.md`。
- `规格索引` 中不存在未完成任务，但下一步仍是 `wf-code-generator` → ❌，建议更新下一步或选择实际待开发任务。
- `代码产出` 中 `T-XXX` 状态为 `✅ 已完成` 但 `output/report-T-XXX.md` 缺失 → ⚠️。
- `测试记录` 中 `T-XXX` 状态为 `✅ 已完成`，但 `output/test-report-T-XXX.md` 缺失 → ⚠️。
- `规格索引` 中 `T-XXX` 标记为 `✅ 已测试` 时，只有该行同时包含对应任务号的 `output/test-report-<任务号>.md`（如 `T-001` 对应 `output/test-report-T-001.md`）才视为规格索引测试状态自洽；否则依赖 `## 测试记录` 表判断测试完成状态。
- 判断是否有待测试任务时，使用 `代码产出` 中已完成任务减去 `测试记录` 中已完成任务；集合非空才建议执行 `wf-test-generator`。
- 没有已完成但未测试任务，但下一步仍是 `wf-test-generator` → ⚠️；如果规格索引中至少存在一个 `T-XXX`，且全部任务均已标记 `✅ 已测试`，建议将阶段更新为 `测试完成`。

#### 问题编号规则

如果 `ISSUES.md` 不存在，跳过本检查项，仅在健康检查说明中标记文件缺失。

如果 `ISSUES.md` 存在，按 markdown 二级标题区间解析 Q 条目：从某个 `## 阶段标题` 开始，到下一个 `## ` 标题之前，均归属该阶段。

- 同一问题编号重复 → ⚠️。
- Q 条目不在 `## 分析阶段`、`## 设计阶段`、`## 实现阶段`、`## 测试阶段` 任一标题下 → ⚠️。
- 除明确包含 `状态：已解决` 的 Q 条目外，`ISSUES.md` 中存在任何 Q 条目即视为待决策问题 → ⚠️，但不默认阻塞下一步。
- 兼容旧工作空间：明确包含 `状态：已解决` 的 Q 条目不计入待决策数，但标记 ⚠️，建议归档到 `CHANGELOG.md` 并从 `ISSUES.md` 删除。

#### 状态分级

| 状态 | 含义 |
|------|------|
| ✅ | 未发现结构问题 |
| ⚠️ | 发现不一致或待决策项，可能需要人工处理 |
| ❌ | 缺少下一步必需输入，或当前状态与下一步不一致，继续执行大概率失败 |

### Step 3：输出状态

```
# 工作空间状态

项目：XXX | 平台：XXX | 扫描时间：YYYY-MM-DD HH:MM

## 进度

当前阶段：XXX
下一步：XXX

## 待决策

如果 `ISSUES.md` 存在，使用以下模板：

| 阶段 | 待决策数 |
|------|---------|
| 分析阶段 | X |
| 设计阶段 | X |
| 实现阶段 | X |
| 测试阶段 | X |

> 共 X 项待人工决策，详见 ISSUES.md

如果 `ISSUES.md` 缺失，使用以下模板：

> 待决策数量未知（ISSUES.md 缺失，未扫描）

## 健康检查

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 工作空间文件 | ✅ | 必要文件完整 |
| PRD 文件 | ✅ | prd/ 中存在 X 个支持格式文件 |
| 阶段产物 | ❌ | 下一步是 wf-code-generator，但规格索引中没有未完成任务 |
| 任务索引 | ⚠️ | T-003 缺少 output/specs/T-003.md |
| 问题编号 | ⚠️ | 实现阶段存在 2 个待决策项 |
```

> 表格内容必须按实际检查结果替换，不得照抄示例状态。

### Step 4：输出建议

简短建议（不超过两句话）：

建议按以下优先级输出。主建议只输出最高优先级；详细检查模式列出全部问题。

1. 存在 ❌：指出最关键的阻塞项；若是缺输入则建议补齐，若是状态不一致则建议修正 `CONTEXT.md`。
2. 存在非待决策的 ⚠️ 一致性问题：指出最影响下一步的问题，建议先修正工作空间状态。
3. 有待决策项：指出哪个阶段待决策最多，建议优先处理。
4. 无未决策项且无一致性问题：按工作流顺序推荐下一个未完成的阶段。

### 详细检查模式

用户明确说「详细检查」「严格检查」「健康检查」「validate」时，在上述输出后追加详细问题列表：

有详细问题时使用以下模板：

```
## 详细问题

| 严重级别 | 类型 | 问题 | 建议 |
|----------|------|------|------|
| ⚠️ | 任务索引 | CONTEXT.md 中 T-003 缺少 output/specs/T-003.md | 重新执行 wf-spec-generator 或修正 CONTEXT.md |
```

没有详细问题时使用以下模板：

```
## 详细问题

未发现详细一致性问题。
```
