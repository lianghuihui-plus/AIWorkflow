# 技术方案设计

拆解需求为开发任务，结合代码库进行技术设计，产出 `output/design.md`。

> 在 workspace 目录下执行。

## 前置检查

当前目录存在 `CONTEXT.md`。不存在 → 报错终止。

## 编号规则

| 编号 | 格式 | 说明 |
|------|------|------|
| 任务 ID | T-001、T-002… | 按生成顺序分配 |
| 需求引用 | REQ-xxx | 保持 analysis.md 中的编号不变 |

---

## 执行流程

### Step 1：读取输入

读 `CONTEXT.md`，提取平台、代码路径。

读 `output/analysis.md`。不存在 → 报错：**「未找到 analysis.md，请先执行 wf-prd-analyzer」**，终止。

提取每条功能需求的完整信息：所属模块、变更内容、处理逻辑、约束条件、需求来源。

### Step 2：读取已有设计

`output/design.md` 存在 → 读取，提取已有任务拆解、模块架构、接口定义，作为后续设计的参考。

不存在 → 跳过。

### Step 3：任务拆解

对每条已纳入的需求拆成任务块：

**拆解粒度参考：**
- 一个独立模块的实现 → 一个任务
- 一个页面的交互逻辑 → 一个任务
- 一个接口/数据通道的对接 → 一个任务
- 一段独立的业务规则 → 一个任务

**拆分信号（满足任一即拆）：** 涉及不同模块或页面 / 可独立开发和验证 / 明显有先后依赖 / 同一需求内涉及多个独立关注点。

**合并信号（满足任一可合并）：** 同一模块/页面的多个关联改动 / 共享同一段代码路径 / 互相不可独立验证。

### Step 4：依赖分析与排序

- **硬依赖**：B 需要 A 的代码产出才能工作 → 在依赖列写明 `T-xxx`
- **软依赖**：B 和 A 代码无耦合 → 不设为依赖
- 无依赖的排前面 → 被依赖的次之 → 同优先级按需求编号顺序

### Step 5：代码库探索

如有代码路径，探索项目结构：扫描顶层 + 一级子目录，按任务关键词匹配模块。

代码路径为空 → 跳过。

### Step 6：逐任务技术设计

对每个任务，基于 Step 5 探索到的代码库实际结构和已有模式进行设计。**保证可实施性：接口定义必须精确到文件、签名、类型；模块划分遵循代码库已有模式。**

- **模块架构** — 涉及哪些模块，模块职责和关系。遵循代码库已有的模块划分模式
- **接口/方法定义** — 精确到：哪个文件、什么签名、入参出参类型。例如 `src/services/AuthService.ets` 中新增 `login(params: LoginParams): Promise<LoginResult>`
- **数据结构** — 新增或修改的 class/struct/interface，字段和类型
- **约束条件落地** — 将需求中的约束条件转化为设计约束。如"接口 3 秒超时"→ 接口定义中标注超时机制；"支持取消"→ 设计取消回调
- **影响范围** — 需修改的文件（目录级）、依赖模块、可能受影响的已有功能

设计过程中的不确定项 → 标注（Step 8 统一写入 ISSUES.md）。

### Step 7：生成 design.md

`output/` 目录不存在则先创建。

```markdown
# 技术方案 — [项目名称]

> 设计日期：YYYY-MM-DD
> 来源：analysis.md

---

## 任务总览

| ID | 任务标题 | 关联需求 | 依赖 | 状态 |
|----|---------|---------|------|------|
| T-001 | 登录接口接入 | REQ-001 | — | 待开发 |
| T-002 | 登录页交互开发 | REQ-001 | T-001 | 待开发 |

---

## 任务详情

### T-001 — 登录接口接入

**关联需求：** REQ-001

**需求摘要：**
- 核心逻辑：手机号+验证码登录接口对接
- 校验规则：手机号 11 位数字格式校验
- 交互约束：验证码 60 秒倒计时，接口 3 秒超时

**依赖：** 无

**模块架构：**
- 登录模块负责认证流程，复用项目已有的 Service 模式

**接口/方法定义：**
- `src/services/AuthService.ets` — `login(params: LoginParams): Promise<LoginResult>`

**数据结构：**
- 新增 `LoginParams`（phone: string, code: string），`LoginResult`（token: string, userId: string）

**影响范围：**
- 新增：`src/api/auth.ets`
- 修改：`src/services/AuthService.ets`
- 依赖：http 工具类
```

### Step 8：更新工作空间文件

**更新 CONTEXT.md：**

- `## 当前状态` → `阶段` 改为 `技术方案完成，待审核`，`下一步` 改为 `执行 wf-spec-generator`

**写入 ISSUES.md（Step 6 中的不确定项）：**

在 `## 设计阶段` 下按以下格式逐条追加，编号按已有 Q-xxx 递增：

```
### Q-001 — [问题简述]

- **问题：** [具体描述]
- **AI 建议：** [建议方案]
- **影响：** [影响的产物或模块]
- **提出：** YYYY-MM-DD wf-tech-designer

---

**人工决策：**
**状态：** 待决策
```

**写入 JOURNAL.md：**

在文件末尾追加当天日期标题（如已存在则复用），然后追加：

```
### HH:MM — wf-tech-designer 完成

- 拆解 X 个任务，完成技术方案设计
- 生成 output/design.md
- 发现问题 X 项，写入 ISSUES.md
```
