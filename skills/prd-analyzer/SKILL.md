# PRD 需求分析

从原始 PRD 文档中提取结构化需求，产出 `requirements.md`。

## 配置读取

**本阶段所有输入从 `workflow.yaml` 读取，不交互询问。必须在 workflow-xxx 目录下执行。**

### 从 workflow.yaml 读取

| 配置路径 | 用于 | 说明 |
|---------|------|------|
| `prd.sources` | PRD 文件列表 | 必填。如为空，终止并告知用户 |
| `project.platform` | 目标平台 | 必填。如为空，终止并告知用户 |
| `project.code_path` | 项目代码路径 | 可选。为空则跳过项目结构扫描 |
| `output.base_dir` | 输出根目录 | 必填。默认 `output` |
| `project.name` | 项目名称 | 用于文档标题 |

### 路径规则

- `workflow.yaml` 中所有路径均为相对路径，以 workflow-xxx 根目录为基准
- PRD 文件路径如不存在，**原样记录不存在的文件并跳过**（不中断流程），在需求分析报告中列出缺失的文件

## 输出

写入 `{output.base_dir}/requirements.md`，结构见下方模板。

## 工作流程

### 模式判断

校验必填字段（`prd.sources`、`project.platform`）：
- 如为空 → 终止并告知用户：**「workflow.yaml 中缺少必填配置，请检查后重试」**

读取 `workflow.yaml` 获取 `output.base_dir`（默认 `output`），记为 `OUT`。检查 `{OUT}/requirements.md` 是否存在：

**文件不存在** → 进入「完整流程」

**文件存在** → 读取文档状态：
- 状态为「已确认」→ 告知用户：**「需求文档已完成（当前 vX.X），所有待澄清项已解决，无需修订」**，终止
- 状态为「待审核」→ 进入「修订模式」

## 编号规则

| 编号类型 | 格式 | 分配规则 |
|---------|------|---------|
| PRD ID | PRD-01、PRD-02… | 按 `prd.sources` 顺序分配 |
| 需求 ID | REQ-001、REQ-002… | 全部分配 REQ 编号，纳入和暂不纳入共用，按检出顺序 |
| 待澄清 ID | CL-001、CL-002… | 按检出顺序分配 |
| 原文引用锚点 | A-001↔REQ-001, C-001↔REQ-001 | 与需求 ID 一一对应，暂不纳入的引用也用同 ID 的 C 锚点 |

- 各表中引用用 `[A-001]` 格式，附录中 `<a id="A-001">` 定义锚点
- 来源列统一使用 PRD ID，完整文件名映射见「PRD 来源」表

## 完整流程（初版生成）

### Step 1：读取 PRD

逐份读取 `prd.sources` 中的所有 PRD 文档，**全量提取，不做任何判断和过滤**：

- **为每份 PRD 按顺序分配 ID**（PRD-01、PRD-02…），记录完整文件名和标题，写入「PRD 来源」表
- 标注每条需求的来源 PRD ID
- 文件清单中使用 PRD ID（PRD-01…），不再重复完整文件名；缺失的 PRD 也在文件清单中列出
- 摘录原文关键句，按编号规则分配引用锚点。暂不纳入的需求也分配 C 锚点，ID 仍为 REQ

文件不存在的 → 在「文件清单」中标记为 ⚠️ 缺失，继续处理其他文件。

### Step 2：统一分流

对 Step 1 提取的每条需求逐条判断。如项目代码路径存在：

1. 先扫项目根目录结构（顶层 + 一级子目录）
2. 按需求中的领域关键词（模块名、业务词）匹配对应目录
3. 读匹配模块的入口或主要文件辅助判断

分流规则：

- 属于目标平台 → 写入「功能需求」
- 不属于目标平台 → 写入「暂不纳入」，标注原因
- **无法判断 → 放入「待澄清」**

代码扫描仅辅助判断模块归属，不改变分流结果。

### Step 3：结构化补全与生成

对**已纳入的功能需求**逐条补全。每条需求必须包含以下字段：
- 所属模块 — 模块名/页面/组件
- 变更内容 — 按条目列出，每条一句话说明要做什么。行为型需求按场景→流程顺序，变更型需求逐项列出
- 处理逻辑 — 按条目列出，每条说明一项校验规则、判断条件或状态变化
- 约束条件 — 多项时逐条列出
- 需求来源 — PRD ID
- 需澄清 — PRD 未提及但实现必需的信息，用表格列出（序号/问题/AI建议/人工回应/AI复核/状态）

局部约束和全局约束统一写入「约束条件」字段。
某个字段信息缺失 → 在对应字段中标注 `[待确认：xxx]`，同时在需求小节的「需澄清」表中新增条目，初始状态为「待处理」。

**深度检测：** 识别以下问题并写入独立的「待澄清」表：
- 流程描述模糊或存在歧义
- 不同 PRD 对同一需求描述矛盾
- 缺少关键信息（交互细节、数据定义、异常处理等）

**规则：纳入 ≠ 没问题。纳入只代表平台匹配，待澄清负责找已纳入项的雷。**

- 纳列入厂留空（填 Y / N）
- 附录只保留 A（功能需求原文）和 C（暂不纳入原文）两个分区

### REQ-001 — 手机号验证码登录

**所属模块：** 登录模块-登录页面

**变更内容：**
- 新增手机号+验证码登录入口
- 进入登录页后，用户输入手机号和 6 位验证码，点击登录按钮发起请求
- 校验通过后发送登录请求，3 秒超时
- 成功：跳转首页，清除验证码倒计时

- 手机号非 11 位数字时立即提示，不发送请求

## 修订模式

### Step 1：处理审核

读取 `{OUT}/requirements.md`，收敛三层：

**层级一：纳入判断（一览表「纳入」列）**
- 纳入 = Y → 标记该 REQ 为「已确认」
- 纳入 = N → 该 REQ 从一览表和小节中移除，新增到「暂不纳入」，保持原 REQ ID 不变
- 纳入为空 → 保持

**层级二：需澄清（需求小节内表格）**
- 扫描每条需求下「需澄清」表的「人工回应」列
- 有内容 → AI 复核其是否解决了原问题：
  - 已明确解决 → 合并回需求描述对应字段，状态改「已解决」
  - 不清晰或不完整 → 状态改「需补充」，在 AI 复核列附说明
- 人工回应为空 → 保持「待处理」

**层级三：待澄清（独立表格）**
- 扫描「人工决策」列
- 有内容 → AI 复核：
  - 已明确解决 → 按决策更新对应需求，CL 状态改「已解决」
  - 不清晰或不完整 → CL 状态改「需补充」，AI 复核列附说明
- 人工决策为空 → 保持「待处理」

用户消息中的额外审核意见一并处理。已解决/已确认条目不再处理，除非审核人明确要求。

### Step 2：追加修订记录

在「修订记录」表格中追加一行：

| 日期 | 版本 | 修改内容 |
|------|------|---------|
| YYYY-MM-DD | vX.Y | 简述本轮修改内容 |

版本号读取现有记录最后一行递增，无记录时从 v1.0 开始。

### Step 3：检查完成标准

三层全部满足才改文档状态为「已确认」：
- 所有功能需求的「纳入」列均为 Y
- 所有需求小节的「需澄清」表均无未解决项（状态全为「已解决」）
- 所有 CL 条目的「状态」均为「已解决」

任一项未满足 → 保持「待审核」。
