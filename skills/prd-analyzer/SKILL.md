# PRD 需求分析

从原始 PRD 文档中提取结构化需求，产出 `requirements.md`。

## 配置读取

### 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

定位到 `workflow.yaml` 后，立即解析并执行：

**除工作流目录定位外，本阶段所有输入从 `workflow.yaml` 读取，不交互询问。**

### 读取并校验

1. 解析 `workflow.yaml`，校验 YAML 格式
2. 校验必填字段：
   - `prd.sources` — 如为空 → 终止并告知用户：**「workflow.yaml 中缺少必填配置，请检查后重试」**
   - `project.platform` — 如为空 → 终止并告知用户
3. 提取可选字段：
   - `project.code_path` — 为空则跳过项目结构扫描
   - `project.name` — 用于文档标题
4. 提取 `output.base_dir`（默认 `output`），记为 `OUT`

YAML 格式非法导致无法解析 → 终止并告知用户具体错误。后续步骤直接使用上述变量，不再重复读取。

### 配置字段说明

| 配置路径 | 含义 | 必填 |
|---------|------|------|
| `prd.sources` | PRD 文件列表 | 是 |
| `project.platform` | 目标平台 | 是 |
| `project.code_path` | 项目代码路径 | 否 |
| `output.base_dir` | 输出根目录 | 是（默认 `output`） |
| `project.name` | 项目名称 | 否 |

## 编号规则

| 编号类型 | 格式 | 分配规则 |
|---------|------|---------|
| PRD ID | PRD-01、PRD-02… | 按 `prd.sources` 顺序分配 |
| 需求 ID | REQ-001、REQ-002… | 全部分配 REQ 编号，纳入和暂不纳入共用，按检出顺序 |
| 需澄清 ID | C-REQXXX-N | 归属该需求，独立编号，如 C-REQ001-1 |
| 待澄清 ID | CL-001、CL-002… | 按检出顺序分配 |
| 原文引用锚点 | A-001↔REQ-001, C-001↔REQ-001 | 与需求 ID 一一对应，暂不纳入的引用也用同 ID 的 C 锚点 |

- 各表中引用用 `[A-001]` 格式，附录中 `<a id="A-001">` 定义锚点
- 来源列统一使用 PRD ID，完整文件名映射见「PRD 来源」表

## 工作流程

输出写入 `{output.base_dir}/requirements.md`，结构见下方模板。

### 模式判断

检查 `{OUT}/requirements.md` 是否存在：

**文件不存在** → 进入「完整流程」

**文件存在** → 读取文档状态：
- 状态为「已确认」→ 告知用户：**「需求文档已完成（当前 vX.X），所有待澄清项已解决，无需修订」**，终止
- 状态为「待审核」→ 进入「修订模式」
- 状态为其他值或无法解析 → 视为「待审核」，进入修订模式，并在操作提示中标注异常状态值

> 任何步骤遇到不可恢复的错误（如输出目录不可写），终止并将错误信息告知用户，不静默跳过。

### 完整流程（初版生成）

#### Step 1：读取 PRD

逐份读取 `prd.sources` 中的所有 PRD 文档，**全量提取，不做任何判断和过滤**：

- **为每份 PRD 按顺序分配 ID**（PRD-01、PRD-02…），记录完整文件名和标题，写入「PRD 来源」表
- 标注每条需求的来源 PRD ID
- 文件清单中使用 PRD ID（PRD-01…），不再重复完整文件名；缺失的 PRD 也在文件清单中列出
- 摘录原文关键句，按编号规则分配引用锚点。暂不纳入的需求也分配 C 锚点，ID 仍为 REQ

文件不存在的 → 在「文件清单」中标记为 ⚠️ 缺失，继续处理其他文件。
文件存在但内容为空 → 在「文件清单」中标记为 ⚠️ 空文件，跳过该文件。
`prd.sources` 中存在重复路径 → 去重后处理，在文件清单中标注。

#### Step 2：统一分流

对 Step 1 提取的每条需求逐条判断。如项目代码路径存在：

1. 先扫项目根目录结构（顶层 + 一级子目录）
2. 按需求中的领域关键词（模块名、业务词）匹配对应目录
3. 读匹配模块的入口或主要文件辅助判断

分流规则：

- 属于目标平台 → 写入「功能需求」
- 不属于目标平台 → 写入「暂不纳入」，标注原因
- **无法判断 → 放入「待澄清」**

代码扫描仅辅助判断模块归属，不改变分流结果。

#### Step 3：结构化补全与生成

对**已纳入的功能需求**逐条补全。每条需求必须包含以下字段：
- 所属模块 — 模块名/页面/组件
- 变更内容 — 按条目列出，每条一句话说明要做什么。行为型需求按场景→流程顺序，变更型需求逐项列出
- 处理逻辑 — 按条目列出，每条说明一项校验规则、判断条件或状态变化
- 约束条件 — 多项时逐条列出
- 需求来源 — PRD ID
- 需澄清（REQ 内问题）— 与该 REQ 相关但 PRD 未明确的问题，包括字段缺失、流程模糊、关键信息缺失等。用总览表加逐条段落格式（总览表含 ID/问题/状态，逐条段落含 关联需求/AI建议/人工回应/AI复核/状态）。ID 格式：C-REQXXX-N（C=Clarification，REQXXX=归属需求，N=该需求内序号）

局部约束和全局约束统一写入「约束条件」字段。
某个字段信息缺失 → 在对应字段中标注 `[待确认：xxx]`，同时在需求小节的「需澄清」总览表和段落中各新增一条，初始状态为「待处理」。

**深度检测：** 逐条扫描已纳入的需求，按问题归属分流：
- 仅影响单条 REQ → 写入该 REQ 小节的「需澄清」表（字段缺失、流程模糊、缺关键信息等）
- 跨多条 REQ 或跨 PRD → 写入独立的「待澄清（跨 REQ / 全局）」表（PRD 矛盾、需求冲突、功能范围不确定等）

**规则：纳入 ≠ 没问题。纳入只代表平台匹配，「需澄清」和「待澄清」各司其职找雷，同一条问题不重复出现在两个表。**

完成补全后，按下方模板生成 `{OUT}/requirements.md`：

**生成规则：**
- 所有来源列使用 PRD ID（PRD-01、PRD-02…），不使用完整文件名
- 纳入列留空（填 Y / N）
- 附录只保留 A（功能需求原文）和 C（暂不纳入原文）两个分区

### 修订模式

#### Step 1：处理审核

读取 `{OUT}/requirements.md`，收敛三层：

**层级一：纳入判断（一览表「纳入」列）**
- 纳入 = Y → 标记该 REQ 为「已确认」
- 纳入 = N → 该 REQ 从一览表和小节中移除，新增到「暂不纳入」，保持原 REQ ID 不变。同时：
  - 将该 REQ 的原文引用锚点从「附录 A — 功能需求原文」迁移到「附录 C — 暂不纳入原文」（`<a id="A-xxx">` → `<a id="C-xxx">`），同步更新一览表中的引用列
  - 该 REQ 下的所有 C-REQxxx-N 需澄清条目同步关闭，在 CL 表追加一条说明：「REQ-xxx 整体不纳入，其下 N 条需澄清同步关闭」，状态标记为「已解决（不纳入）」
- 纳入为空 → 保持

**层级二：需澄清（需求小节内）**
- 扫描每条需求下「需澄清」总览表的「状态」列
- 状态为「待处理」→ 读取对应段落的「人工回应」字段
- 人工回应有内容 → AI 复核其是否解决了原问题：
  - 已明确解决 → 按以下映射将澄清内容补入需求描述对应字段，同时填写「AI 复核」字段说明合并结果，状态改「已解决」。**⚠️ 已解决的 C-REQ 条目和详情块必须保留在文档中，不得删除。**

    | 问题类型 | 合并目标字段 | 示例 |
    |---------|------------|------|
    | 字段缺失（如缺超时值、缺返回字段） | 约束条件 | 「接口超时 3 秒」→ 补入约束条件 |
    | 流程模糊（如缺步骤、缺分支） | 变更内容 | 「失败后跳转回登录页」→ 补入变更内容 |
    | 校验规则缺失（如格式校验、边界） | 处理逻辑 | 「手机号仅支持中国大陆号段」→ 补入处理逻辑 |
    | 范围/权限不明确 | 约束条件 | 「仅管理员可见」→ 补入约束条件 |
    | 无法明确归类 | 约束条件 | 默认归入约束条件 |

  - 不清晰或不完整 → 保持「待处理」，AI 复核字段附说明（告知用户哪些点仍需澄清）
- 人工回应为空 → 保持「待处理」

**层级三：待澄清（独立段落）**
- 扫描「待澄清」总览表的「状态」列
- 状态为「待处理」→ 读取对应段落的「人工决策」字段
- 人工决策有内容 → AI 复核：
  - 已明确解决 → 按以下规则更新对应需求，同时填写「AI 复核」字段说明处理结果，CL 状态改「已解决」。**⚠️ 已解决的 CL 条目和详情块必须保留在文档中，不得删除。**

    | 决策类型 | 更新方式 |
    |---------|---------|
    | 采纳方案 A / 确定字段定义 | 更新受影响 REQ 的对应字段（变更内容/约束条件/处理逻辑），遵循层级二的映射规则 |
    | 确定纳入范围（新增 REQ） | 在「功能需求」一览表追加新 REQ，补全结构化字段，从原 PRD 摘录原文到附录 A |
    | 确定不纳入 | 将相关 REQ 移入「暂不纳入」，按层级一规则处理锚点和澄清项 |
    | 仅补充说明 | 在相关 REQ 的约束条件或处理逻辑中追加说明文字 |

  - 不清晰或不完整 → 保持「待处理」，AI 复核字段附说明（告知用户哪些点仍需澄清）
- 人工决策为空 → 保持「待处理」

扫描用户消息中提到的 REQ ID、C-REQ ID 或 CL ID，将对应审核意见应用到匹配条目；无明确 ID 时按关键词匹配。已解决/已确认条目不再处理，除非审核人明确要求。

#### Step 2：追加修订记录

在「修订记录」表格末尾追加一行（新条目始终追加在表格最后一行之后，不在顶部插入）。版本号读取现有记录最后一行递增，无记录时从 v1.0 开始。

| 日期 | 版本 | 修改内容 | 执行耗时 |
|------|------|---------|--------|
| YYYY-MM-DD | vX.Y | 简述本轮修改内容 | X 秒 / X 分钟 X 秒 |

#### Step 3：检查完成标准

三层全部满足才改文档状态为「已确认」：
- 所有功能需求的「纳入」列均为 Y
- 所有需求小节的「需澄清」总览表均无未解决项（状态全为「已解决」）
- 所有待澄清 CL 条目的「状态」均为「已解决」

任一项未满足 → 保持「待审核」。
