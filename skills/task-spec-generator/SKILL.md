# 开发规格生成

从 `tech-design.md` 中读取技术方案，为每个任务生成独立的规格文档和总览索引。

> 本阶段做**格式收敛和细化**：从架构级技术方案收敛为文件/方法级的执行指令。不写完整实现代码，这些留给 code-generator。

## 配置读取

### 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

定位到 `workflow.yaml` 后，立即解析并执行：

**除工作流目录定位外，本阶段所有输入从 `workflow.yaml` 读取，不交互询问。**

### 读取并校验

1. 解析 `workflow.yaml`，校验 YAML 格式
2. 校验必填字段：
   - `project.platform` — 如为空 → 终止并告知用户
3. 提取可选字段：
   - `project.name` — 用于文档标题
4. 提取 `output.base_dir`（默认 `output`），记为 `OUT`
5. 记 `SPECS_DIR` = `{OUT}/specs`

YAML 格式非法导致无法解析 → 终止并告知用户具体错误。后续步骤直接使用上述变量，不再重复读取。

### 配置字段说明

| 配置路径 | 含义 | 必填 |
|---------|------|------|
| `project.platform` | 目标平台 | 是 |
| `output.base_dir` | 输出根目录 | 是（默认 `output`） |
| `project.name` | 项目名称 | 否 |

### 路径规则

- `workflow.yaml` 中所有路径均为相对路径，以 workflow-xxx 根目录为基准
- 所有路径拼接均以 workflow-xxx 根目录为基准：`{OUT}/` 等价于 `{workflow根目录}/{OUT}/`
- 规格文件写入 `{SPECS_DIR}/`，如目录不存在则创建

|------|------|------|
| 任务 ID | T-001、T-002… | 保持 tech-design.md 中的编号不变 |
| 修改点 ID | M-001、M-002… | 每个任务内独立编号，从 M-001 开始 |
| 待澄清 ID | CL-001、CL-002… | 每个任务内独立编号，从 CL-001 开始 |

## 工作流程

输出：
- 每个任务一个规格文件：`{SPECS_DIR}/T-XXX-spec.md`
- 总览索引：`{SPECS_DIR}/README.md`

### 前置检查

检查 `{OUT}/tech-design.md`：
- 不存在 → 终止并告知用户：**「未找到 tech-design.md，请先执行 tech-designer」**
- 存在但「状态」不为「待审核」或「已确认」→ 终止并告知用户：**「技术文档状态异常（当前状态：xxx），请确认后再执行」**

读取 `{OUT}/tech-design.md` 全文，确认至少有一个任务 → 如无任何任务则终止并告知用户。

> 任何步骤遇到不可恢复的错误，终止并将错误信息告知用户，不静默跳过。

---

### 完整流程（初版生成）

#### Step 1：逐任务提取

解析 `tech-design.md`，按任务小节（`## T-XXX — ...`）拆分为独立的任务单元。对每个任务提取：

- 任务标题、关联需求
- 涉及文件（路径 + 操作类型：新增/修改）
- 数据结构变更（字段、类型）
- 核心逻辑流程
- 入口与出口
- 全局约束（如有）
- 边界场景

如果某个任务的方案信息不完整（如缺核心流程、缺数据结构），记录为「待补充」，不强行推断。

#### Step 2：逐规格细化

对每个任务，基于技术方案进一步细化为**逐文件逐修改点**的执行规格：

**每个修改点必须包含：**

| 内容 | 说明 |
|------|------|
| 目标文件 | 真实文件路径（从 tech-design 继承，不确定则标注「需确认」） |
| 操作类型 | `新增文件` / `修改文件` / `删除文件` |
| 修改点描述 | 在这个文件里具体改什么（精确到方法/字段/逻辑） |
| 方法/字段签名 | 新增方法需写明函数签名（参数类型、返回值类型）；新增字段需写明类型和默认值 |
| 前置依赖 | 同一任务内依赖的其他修改点 ID（M-xxx），无依赖填 `—` |
| 验证方式 | 怎么确认这个修改点改对了 |

**细化原则：**
- 一个修改点 = 一个文件中的一个独立改动（新增一个方法 / 新增一个字段 / 修改一段逻辑）
- 如果一个文件有多个改动点，分别列出
- 修改点的排序 = 执行顺序（有依赖的前置依赖必须先完成）
- 对于新增文件，列出完整的文件结构（类定义 + 方法列表 + 字段列表），不需要完整实现
- **不要猜测不存在的类/方法/路径**，如果不确定，标注为「需确认」

#### Step 3：生成规格文件

**生成规则：**
- 每个任务生成一个独立的 `{SPECS_DIR}/T-XXX-spec.md` 文件，状态初始为「待审核」
- 修改点 ID 每个任务内独立编号，从 M-001 开始
- 待澄清 ID 每个任务内独立编号，从 CL-001 开始
- 修改点清单中依赖列：无依赖填 `—`，有依赖填 `M-xxx`
- 同时生成 `{SPECS_DIR}/README.md` 总览索引，状态初始为「待审核」

**单个任务规格文件：**

```markdown
# T-001 — [任务标题]

> 生成日期：YYYY-MM-DD
> 关联需求：REQ-001
> 来源：tech-design.md
> 状态：待审核

## M-002 — 新增登录方法

**目标文件：** `feature/login/src/main/ets/LoginApi.ets`
**操作：** 修改文件

**修改内容：**
- 新增方法 `login(phone: string, code: string): Promise<LoginResponse>`

  1. 校验手机号格式（11 位数字）
  2. 调用后端登录接口

**前置依赖：** M-001（需要先有 LoginResponse 模型）

**验证方式：** 单元测试覆盖正常/异常响应，断言 token 正确返回

## 边界场景

- 网络超时 → 返回错误码，不抛出未捕获异常
- 手机号格式错误 → 在方法入口处拦截，不调网络请求
- token 为空 → 按登录失败处理
```

### 修订模式

本阶段有两套独立的修订流程：

| 流程 | 对象 | 用途 |
|------|------|------|
| A. 索引修订 | `README.md` | 管理任务增删、调整顺序 |
| B. 规格修订 | 单个 `T-XXX-spec.md` | 收敛单个任务的待澄清项 |

> 索引修订时新增/删除任务会同步创建/删除对应的 spec 文件。规格修订只修改单个 spec 文件内容，不修改索引文件。

---

### A. 索引文件修订（README.md）

#### Step A1：模式判断

检查 `{SPECS_DIR}/README.md` 的文档状态：
- 状态为「已确认」→ 告知用户：**「规格索引已完成（当前 vX.X），无需修订」**，终止
- 状态为「待审核」→ 继续处理
- 状态为其他值或无法解析 → 视为「待审核」，继续处理，并在操作提示中标注异常状态值

#### Step A2：索引修订

**新增任务：**
1. 在「规格列表」表中追加一行（标题、修改点数、依赖、状态填「待审核」）
2. 在「规格文件」列表中追加对应的文件链接
3. 创建对应的 `{SPECS_DIR}/T-XXX-spec.md` 文件（状态「待审核」）
4. 在索引的「修订记录」中追加一行

**删除任务：**
1. 从「规格列表」表中删除对应行
2. 从「规格文件」列表中删除对应链接
3. 删除对应的 `{SPECS_DIR}/T-XXX-spec.md` 文件
4. 在索引的「修订记录」中追加一行

#### Step A3：检查完成标准

索引表与实际文件列表一致 → 文档状态改为「已确认」。不一致 → 保持「待审核」。

> 索引修订完成后执行「输出前自检」。

---

### B. 任务规格修订（单个 T-XXX-spec.md）

#### Step B1：模式判断

检查对应规格文件（如 `T-001-spec.md`）的文档状态：
- 状态为「已确认」→ 告知用户：**「该任务规格已完成（当前 vX.X），无需修订」**，终止
- 状态为「待审核」→ 继续处理
- 状态为其他值或无法解析 → 视为「待审核」，继续处理，并在操作提示中标注异常状态值

#### Step B2：处理审核

**待澄清条目：**
- 扫描每个 CL 详情块中的「人工决策」行
- 有内容 → AI 复核：
  - 已明确解决 → 按决策调整规格（更新修改点/新增修改点/补充字段），CL 状态改「已解决」
  - 不清晰或不完整 → CL 状态改「需补充」，AI 复核附说明
- 人工决策为空 → 保持「待处理」

扫描用户消息中提到的 CL ID 或修改点 ID，将对应审核意见应用到匹配条目；无明确 ID 时按关键词匹配。已解决条目不再处理，除非审核人明确要求。

#### Step B3：追加修订记录

在「修订记录」表格中追加一行。版本号读取现有记录最后一行递增，无记录时从 v1.0 开始。

#### Step B4：检查完成标准

所有 CL 条目的「状态」均为「已解决」→ 文档状态改为「已确认」。
任一项未满足 → 保持「待审核」。

> 规格修订完成后执行「输出前自检」。

## 输出前自检

生成或修订后，确认以下全部通过：

- [ ] 每个 tech-design 中的任务都有对应的 `T-XXX-spec.md` 文件
- [ ] 每个修改点都包含 6 项（目标文件/操作类型/描述/签名/前置依赖/验证方式）
- [ ] 修改点 ID 每个任务内从 M-001 开始连续编号
- [ ] 修改点清单中的依赖列与实际前置依赖一致
- [ ] `README.md` 中的规格列表与实际文件一一对应
- [ ] 每个 spec 文件的待澄清总览表与详情块一一对应
- [ ] 修订记录版本号连续，最新一行在最下
- [ ] 文档状态与实际完成标准一致
