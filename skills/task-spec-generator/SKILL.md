# 开发规格生成

从 `tech-design.md` 中读取技术方案，为每个任务生成独立的规格文档。

> 本阶段做**格式收敛和细化**：从架构级技术方案收敛为文件/方法级的执行指令。不写完整实现代码，这些留给 code-generator。

## 输入与输出

## Step 1 — 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

**本阶段除 Step 1 外所有输入从 `workflow.yaml` 读取，不交互询问。**

### 输入（workflow.yaml）

| 配置路径 | 用于 | 说明 |
|---------|------|------|
| `output.base_dir` | 输出根目录 | 必填。默认 `output` |
| `project.name` | 项目名称 | 用于文档标题 |
| `project.platform` | 目标平台 | 必填，如 HarmonyOS / Android / iOS |

### 输出

为每个任务写入独立文件：`{output.base_dir}/specs/T-XXX-spec.md`。
同时写入规格总览：`{output.base_dir}/specs/README.md`。

---

## 工作流程

### 前置检查

读取 `workflow.yaml` 获取配置，记为：
- `OUT`：`output.base_dir`（默认 `output`）
- `SPECS_DIR`：`{OUT}/specs`

检查 `{OUT}/tech-design.md`：
- 不存在 → 终止并告知用户：**「未找到 tech-design.md，请先执行 tech-designer」**
- 存在但「状态」不为「待审核」或「已确认」→ 终止并告知用户：**「技术文档状态异常（当前状态：xxx），请确认后再执行」**

---

### Step 2：逐任务提取

从 `tech-design.md` 中提取每个任务的技术方案：
- 任务标题、关联需求
- 涉及文件（路径 + 操作类型：新增/修改）
- 数据结构变更（字段、类型）
- 核心逻辑流程
- 入口与出口
- 全局约束（如有）
- 边界场景

如果某个任务的方案信息不完整（如缺核心流程、缺数据结构），记录为「待补充」，不强行推断。

---

### Step 3：逐规格细化

对每个任务，基于技术方案进一步细化为**逐文件逐修改点**的执行规格：

**每个修改点必须包含：**

| 内容 | 说明 |
|------|------|
| 目标文件 | 真实文件路径（从 tech-design 继承） |
| 操作类型 | `新增文件` / `修改文件` / `删除文件` |
| 修改点描述 | 在这个文件里具体改什么（精确到方法/字段/逻辑） |
| 方法/字段签名 | 新增方法需写明函数签名（参数类型、返回值类型）；新增字段需写明类型和默认值 |
| 前置依赖 | 这个修改点依赖的其他修改点 ID（同一任务内）或跨任务依赖 |
| 验证方式 | 怎么确认这个修改点改对了 |

**细化原则：**
- 一个修改点 = 一个文件中的一个独立改动（新增一个方法 / 新增一个字段 / 修改一段逻辑）
- 如果一个文件有多个改动点，分别列出
- 修改点的排序 = 执行顺序（有依赖的前置依赖必须先完成）
- 对于新增文件，列出完整的文件结构（类定义 + 方法列表 + 字段列表），不需要完整实现
- **不要猜测不存在的类/方法/路径**，如果不确定，标注为「需确认」

## M-002 — 新增登录方法

**目标文件：** `feature/login/src/main/ets/LoginApi.ets`
**操作：** 修改文件

**修改内容：**
- 新增方法 `login(phone: string, code: string): Promise<LoginResponse>`
- 处理逻辑：
  1. 校验手机号格式（11 位数字）
  2. 调用后端登录接口
  3. 解析响应体中的 token，存入 LoginResponse

**前置依赖：** M-001（需要先有 LoginResponse 模型）

**验证方式：** 单元测试覆盖正常/异常响应，断言 token 正确返回

## 边界场景

- 网络超时 → 返回错误码，不抛出未捕获异常
- 手机号格式错误 → 在方法入口处拦截，不调网络请求
- token 为空 → 按登录失败处理

## 编号规则

## 修订模式

本阶段有两套独立的修订流程：**索引文件修订**（管理任务增删）和**任务规格修订**（单文件收敛）。

---

### A. 索引文件修订（README.md）

**适用场景**：新增任务、删除任务、调整任务顺序。

#### 模式判断

检查 `{SPECS_DIR}/README.md` 的「文档状态」：
- 「已确认」→ 无索引变更需求，无需修订
- 「待审核」→ 进入索引修订流程

#### 索引修订流程

**新增任务：**
1. 在「规格列表」表中追加一行（标题、修改点数、依赖、状态）
2. 在「规格文件」列表中追加对应的文件链接
3. 创建对应的 `{SPECS_DIR}/T-XXX-spec.md` 文件（状态「待审核」）
4. 在索引的「修订记录」中追加一行

**删除任务：**
1. 从「规格列表」表中删除对应行
2. 从「规格文件」列表中删除对应链接
3. 删除对应的 `{SPECS_DIR}/T-XXX-spec.md` 文件
4. 在索引的「修订记录」中追加一行

**完成标准：** 索引表与实际文件列表一致 → 文档状态改为「已确认」。

---

### B. 任务规格修订（单个 T-XXX-spec.md）

**适用场景**：单个任务的规格待澄清收敛。

#### 模式判断

检查对应规格文件（如 `T-001-spec.md`）的文档状态：
- 状态为「已确认」→ 告知用户：**「该任务规格已完成（当前 vX.X），无需修订」**，终止
- 状态为「待审核」→ 进入修订流程

#### 处理审核

**待澄清条目：**
- 扫描每个 CL 详情块中的「人工决策」行
- 有内容 → AI 复核：
  - 已明确解决 → 按决策调整规格，CL 状态改「已解决」
  - 不清晰或不完整 → CL 状态改「需补充」，AI 复核附说明
- 人工决策为空 → 保持「待处理」

#### 追加修订记录

在「修订记录」表格中追加一行。版本号读取现有记录最后一行递增，无记录时从 v1.0 开始。

#### 检查完成标准

所有 CL 条目的「状态」均为「已解决」→ 文档状态改为「已确认」。
任一项未满足 → 保持「待审核」。

> **注意**：任务规格修订不修改索引文件。如果修订导致任务需要删除/新增，通过索引修订流程处理。
