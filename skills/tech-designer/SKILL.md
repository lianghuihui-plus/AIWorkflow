# 技术方案设计

从 `tasks.md` 中读取已分解的任务，结合代码库实际结构，产出每个任务的技术方案文档。

> 本阶段做**技术层面的设计决策**：数据怎么流转、结构怎么改、影响面在哪。不写具体实现代码，不精确定位文件路径，这些留给 task-spec-generator 和 code-generator。

## 配置读取

### 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

定位到 `workflow.yaml` 后，立即解析并执行：

**除工作流目录定位外，本阶段所有输入从 `workflow.yaml` 读取，不交互询问。**

### 读取并校验

1. 解析 `workflow.yaml`，校验 YAML 格式
2. 校验必填字段：
   - `project.code_path` — 如为空 → 终止并告知用户：**「workflow.yaml 中缺少 project.code_path，无法探索代码库」**
   - `project.platform` — 如为空 → 终止并告知用户
3. 提取可选字段：
   - `project.name` — 用于文档标题
4. 提取 `output.base_dir`（默认 `output`），记为 `OUT`
5. 记 `CODE_PATH` = `project.code_path`，`PLATFORM` = `project.platform`

YAML 格式非法导致无法解析 → 终止并告知用户具体错误。后续步骤直接使用上述变量，不再重复读取。

### 配置字段说明

| 配置路径 | 含义 | 必填 |
|---------|------|------|
| `project.code_path` | 代码库路径 | 是 |
| `project.platform` | 目标平台 | 是 |
| `output.base_dir` | 输出根目录 | 是（默认 `output`） |
| `project.name` | 项目名称 | 否 |

## 工作流程

输出写入 `{OUT}/tech-design.md`，结构见下方模板。

### 前置检查

检查 `{OUT}/tasks.md`：
- 不存在 → 终止并告知用户：**「未找到 tasks.md，请先执行 task-decomposer」**
- 存在但「状态」不为「已确认」→ 终止并告知用户：**「任务文档尚未确认（当前状态：xxx），请先完成审核收敛」**

检查 `{OUT}/requirements.md`：
- 不存在 → 终止并告知用户：**「未找到 requirements.md，无法扫描全局约束」**

读取 `{OUT}/tasks.md` 和 `{OUT}/requirements.md` 全文，后续步骤基于其内容设计方案。

> 任何步骤遇到不可恢复的错误，终止并将错误信息告知用户，不静默跳过。

### 模式判断

检查 `{OUT}/tech-design.md` 是否存在：

**文件不存在** → 进入「初版模式」

**文件存在** → 读取文档状态：
- 状态为「已确认」→ 告知用户：**「技术方案已确认（vX.X）」**，询问：**「是否需要更新？（上游任务/需求可能有变更）」**
  - 用户说「更新」→ 进入「更新模式」
  - 用户说「不」→ 终止
- 状态为「待审核」→ 进入「修订模式」
- 状态为其他值或无法解析 → 视为「待审核」，进入修订模式

### 初版模式

#### Step 1：代码探索

对每个任务，根据其需求摘要中的关键词（模块名、类名、字段名、入口名），在 `CODE_PATH` 中探索真实代码结构：

**探索目标：**
- 找到涉及的业务模块对应的代码目录
- 了解现有数据结构和命名规范
- 识别相关的数据流入口和出口
- 确认相关类/组件/页面是否存在，了解其结构

**探索原则：**
- **必须基于真实代码**，不能凭空编造文件路径或类名
- 优先读取目录结构和文件列表确认存在，再读取关键文件内容理解逻辑
- 如果某个概念在代码库中找不到直接对应，记录为「待澄清」，不要强行关联

**探索方法（按平台）：**
- HarmonyOS（ArkTS）：按 `features/模块名/src/main/ets/` 结构查找，关注 `.ets` 文件
- Android（Kotlin/Java）：按 `module/src/main/java/包名/` 结构查找，关注 `.kt`/`.java` 文件
- 其他平台：按项目约定的模块目录结构类推

**探索深度：**
- 找到相关模块目录 → 读取 Index/入口文件了解整体结构
- 找到核心类 → 读取类定义了解字段和方法
- 不需要完整阅读所有实现代码，关注**结构和接口**即可
- **如果代码库过大**：先根据任务关键词精确定位相关子目录，不要全局遍历

#### Step 2：扫描全局约束

检查 `requirements.md` 中已纳入需求的全局约束（识别规则与 task-decomposer 的扫描全局约束步骤一致），记录哪些任务受影响。在全局约束影响下思考技术实现的影响：

- 全局约束对技术实现的影响（如「覆盖所有编程任务」→ 需要在公共层处理而非单点修改）
- 跨模块的架构决策

#### Step 3：技术方案设计

基于代码探索结果和全局约束分析，为每个任务设计技术方案：

**每个任务的技术方案必须包含（按此顺序）：**

| 顺序 | 内容 | 说明 |
|------|------|------|
| 1 | 数据结构变更 | 新增字段、修改字段、新建数据类 |
| 2 | 核心逻辑流程 | 用文字或伪代码描述关键处理流程 |
| 3 | 入口与出口 | 从哪里触发，到哪里结束 |
| 4 | 影响范围 | 哪些现有模块可能受影响 |
| 5 | 边界场景 | 异常、空值、网络失败、并发等 |

> 文件路径的定位和修改点拆解留给 task-spec-generator，本阶段不做。

> 先说改哪里，再说怎么改，最后说注意什么。

**全局约束标记**：如果该任务受全局约束影响，在「数据结构变更」之前插入 `**⚠️ 全局约束：** 描述内容`。

**如果任务涉及多个模块：**
- 按模块分别列出数据结构和设计
- 用架构图或文字描述模块间的调用关系

#### Step 4：生成 tech-design.md

按下方模板生成 `{OUT}/tech-design.md`：

> **任务详情内容排序规则：数据结构 → 核心流程 → 入口与出口 → 影响范围 → 边界场景。**
> 先说改什么，再说怎么改，最后说注意什么。

**生成规则：**
- 任务总览表状态列初稿统一填「待设计」
- 任务 ID 和需求引用保持 tasks.md / requirements.md 中的编号不变
- 任务详情中只保留「关联需求」，不重复「关联任务」
- 全局约束影响的任务在标题后加 `[G]` 标记

> 生成日期：YYYY-MM-DD
> 来源：tasks.md
> 平台：[platform]
> 状态：待审核

## T-001 — 登录接口接入

**关联需求：** REQ-001

### 数据结构变更

**LoginResponse（新增）：**
- `token: string` — 鉴权令牌
- `userId: string` — 用户 ID
- `expiresIn: number` — 过期时间（秒）

**用户信息模型（修改）：**
- 新增 `lastLoginTime: string` 字段

### 核心逻辑流程

1. 用户输入手机号和验证码 → 调用登录页校验逻辑
2. 校验通过 → 调用 `LoginApi.sendCode(phone)` 发送验证码
3. 收到验证码 → 调用 `LoginApi.login(phone, code)`
4. 登录成功 → 保存 token 到本地存储，跳转到首页
5. 登录失败 → 根据错误码提示用户

### 入口与出口

- **入口：** 登录页点击「登录」按钮
- **出口：** 跳转至首页 / 停留在登录页提示错误

### 影响范围

- 影响认证模块的 token 刷新逻辑
- 影响首页的用户信息展示

### 边界场景

- 验证码倒计时期间重复点击 → 按钮置灰
- 网络超时 → 显示重试提示
- Token 过期 → 触发静默刷新或跳转登录页

## T-002 — 登录页交互

**关联需求：** REQ-001

### 数据结构变更

无新增模型，复用 T-001 的 `LoginResponse`。

### 核心逻辑流程

1. 用户进入登录页 → 展示手机号输入框和验证码输入框
2. 用户输入手机号 → 实时校验 11 位数字格式
3. 用户点击「获取验证码」→ 调用 T-001 的发送验证码接口
4. 发送成功 → 按钮置灰，开始 60 秒倒计时
5. 用户输入验证码 + 点击「登录」→ 调用 T-001 的登录接口
6. 登录成功 → 跳转首页；失败 → 展示错误提示，保留已输入内容

### 入口与出口

- **入口：** 启动页 / 个人中心点击登录入口
- **出口：** 登录成功跳转首页 / 失败停留登录页

### 影响范围

- 影响启动页的登录入口跳转逻辑
- 影响登录态过期后的自动跳转

### 边界场景

- 倒计时期间退出页面 → 重新进入时判断倒计时是否结束
- 网络异常 → toast 提示并恢复按钮状态
- 快速切换页面 → 取消未完成的网络请求

## T-003 — 头像上传 [G]

**关联需求：** REQ-002

**⚠️ 全局约束：** 上传能力需覆盖应用内所有需要头像的场景

### 数据结构变更

无新增模型，复用现有 `UploadResponse`。

### 核心逻辑流程

1. 用户点击头像 → 弹出选择（拍照 / 相册）
2. 选择图片 → 校验大小和格式
3. 校验通过 → 调用 `FileUploader.upload(file)`
4. 上传成功 → 更新头像 URL，刷新 UI
5. 上传失败 → 重试或提示

### 入口与出口

- **入口：** 个人主页点击头像
- **出口：** 头像更新完成 / 上传失败提示

### 影响范围

- 影响个人主页、设置页等所有展示头像的位置

### 边界场景

- 图片超过 5MB → 拒绝并提示
- 格式非 jpg/png → 拒绝并提示
- 上传中断（网络断开）→ 支持断点续传

### 修订模式

#### Step 1：处理审核

检查 `{OUT}/tech-design.md` 的文档状态：
- 状态为「待审核」→ 继续处理
- 状态为其他值或无法解析 → 视为「待审核」，继续处理，并在操作提示中标注异常状态值

**待澄清条目：**
- 扫描每个 CL 详情块中的「人工决策」行
- 有内容 → AI 复核：
  - 已明确解决 → 按决策调整方案，CL 状态改「已解决」。**⚠️ 已解决的 CL 条目和详情块必须保留在文档中，不得删除。**
  - 不清晰或不完整 → 保持「待处理」，AI 复核附说明（告知用户哪些点仍需澄清）
- 人工决策为空 → 保持「待处理」

扫描用户消息中提到的 CL ID 或任务 ID，将对应审核意见应用到匹配条目；无明确 ID 时按关键词匹配。已解决条目不再处理，除非审核人明确要求。

#### Step 2：追加修订记录

在「修订记录」表格末尾追加一行（新条目始终追加在表格最后一行之后，不在顶部插入）。版本号读取现有记录最后一行递增，无记录时从 v1.0 开始。

#### Step 3：检查完成标准

所有 CL 条目的「状态」均为「已解决」→ 文档状态改为「已确认」。
任一项未满足 → 保持「待审核」。

> 修订完成后执行「输出前自检」。

### 更新模式

当上游 tasks.md 或 requirements.md 发生变更后，增量更新 tech-design.md。

> 更新模式只重新设计**受影响**的任务。未受影响的任务设计保持不变。

#### Step 1：读取文档

- 读取 `{OUT}/requirements.md` 全文
- 读取 `{OUT}/tasks.md` 全文
- 读取 `{OUT}/tech-design.md` 全文

#### Step 2：识别变更

对比 tasks.md 中的任务与 tech-design.md 中的设计小节：

| 情况 | 判定 | 操作 |
|------|------|------|
| 任务有匹配设计，任务内容未变 | 无变更 | 跳过 |
| 任务有匹配设计，任务需求摘要变化 | 需更新 | 标记「需重新设计」 |
| 任务无匹配设计 | 新增任务 | 标记「待新建设计」 |
| 设计小节对应任务已废弃 | 任务废弃 | 标记「待废弃」 |

#### Step 3：增量更新

**需重新设计：**
- 回到 `CODE_PATH` 重新探索相关代码（如任务变更涉及新模块）
- 按初版生成规则重新生成该任务的设计小节（数据结构 → 核心流程 → 入口出口 → 影响范围 → 边界场景）
- 在任务总览表更新「核心改动」列

**待新建设计：**
- 按初版生成规则（Step 1-3）为新任务生成完整设计小节
- 在任务总览表末尾追加一行
- 状态填「待设计」

**待废弃：**
- 任务状态改为「已废弃」
- 不删除条目

#### Step 4：追加修订记录

在修订记录末尾追加一行：

| 日期 | 版本 | 修改内容 |
|------|------|---------|
| YYYY-MM-DD | vX.Y | 更新：同步上游任务变更。N 个设计更新 / N 个新增 / N 个废弃 |

#### Step 5：状态降级

文档状态改为「待审核」，提示用户：**「技术方案已更新，请审阅后确认」**。

## 输出前自检

生成或修订 `tech-design.md` 后，确认以下全部通过：

- [ ] 所有任务都基于真实代码探索得出结论，未凭空编造路径或类名
- [ ] 每个任务详情按「数据结构 → 核心流程 → 入口出口 → 影响范围 → 边界场景」顺序排列
- [ ] 全局约束影响的任务已标记 `[G]` 或 `⚠️ 全局约束`
- [ ] 任务总览表中的每个任务在详情区都有对应小节
- [ ] 待澄清总览表与详情块一一对应，无孤立的条目
- [ ] 修订记录版本号连续，最新一行在最下
- [ ] 文档状态与实际完成标准一致
