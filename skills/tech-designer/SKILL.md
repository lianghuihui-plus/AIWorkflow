# 技术方案设计

从 `tasks.md` 中读取已分解的任务，结合代码库实际结构，产出每个任务的技术方案文档。

> 本阶段做**技术层面的设计决策**：数据怎么流转、结构怎么改、影响面在哪。不写具体实现代码，不精确定位文件路径，这些留给 task-spec-generator 和 code-generator。

## 输入与输出

## Step 1 — 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

**本阶段除 Step 1 外所有输入从 `workflow.yaml` 读取，不交互询问。**

### 输入（workflow.yaml）

| 配置路径 | 用于 | 说明 |
|---------|------|------|
| `output.base_dir` | 输出根目录 | 必填。默认 `output` |
| `project.name` | 项目名称 | 用于文档标题 |
| `project.platform` | 目标平台 | 必填，如 HarmonyOS / Android / iOS |
| `project.code_path` | 代码库路径 | 必填。相对 workflow.yaml 所在目录，用于探索真实代码 |

### 输出

写入 `{output.base_dir}/tech-design.md`。

## 工作流程

### 前置检查

读取 `workflow.yaml` 获取配置，记为：
- `OUT`：`output.base_dir`（默认 `output`）
- `CODE_PATH`：`project.code_path`（相对于 workflow.yaml 所在目录）
- `PLATFORM`：`project.platform`

检查 `{OUT}/tasks.md`：
- 不存在 → 终止并告知用户：**「未找到 tasks.md，请先执行 task-decomposer」**
- 存在但「状态」不为「待审核」或「已确认」→ 终止并告知用户：**「任务文档状态异常（当前状态：xxx），请确认后再执行」**

检查 `{OUT}/requirements.md`：
- 不存在 → 终止并告知用户：**「未找到 requirements.md，无法扫描全局约束」**

---

### Step 2：代码探索

对每个任务，根据其需求摘要中的关键词（模块名、类名、字段名、入口名），在 `CODE_PATH` 中探索真实代码结构：

**探索目标：**
- 找到涉及的业务模块对应的代码目录
- 了解现有数据结构和命名规范
- 识别相关的数据流入口和出口
- 确认相关类/组件/页面是否存在，了解其结构

**探索原则：**
- **必须基于真实代码**，不能凭空编造文件路径或类名
- 优先读取目录结构和文件列表确认存在，再读取关键文件内容理解逻辑
- 如果某个概念在代码库中找不到直接对应，记录为「待澄清」，不要强行关联

**探索方法（按平台）：**
- HarmonyOS（ArkTS）：按 `features/模块名/src/main/ets/` 结构查找，关注 `.ets` 文件
- Android（Kotlin/Java）：按 `module/src/main/java/包名/` 结构查找，关注 `.kt`/`.java` 文件
- 其他平台：按项目约定的模块目录结构类推

**探索深度：**
- 找到相关模块目录 → 读取 Index/入口文件了解整体结构
- 找到核心类 → 读取类定义了解字段和方法
- 不需要完整阅读所有实现代码，关注**结构和接口**即可
- **如果代码库过大**：先根据任务关键词精确定位相关子目录，不要全局遍历

---

### Step 3：扫描全局约束

检查 `requirements.md` 中已纳入需求的全局约束（与 task-decomposer 的 Step 2.5 一致），记录哪些任务受影响。在全局约束影响下思考技术实现的影响：

- 全局约束对技术实现的影响（如「覆盖所有编程任务」→ 需要在公共层处理而非单点修改）
- 跨模块的架构决策

---

### Step 4：技术方案设计

基于代码探索结果和全局约束分析，为每个任务设计技术方案：

**每个任务的技术方案必须包含（按此顺序）：**

| 顺序 | 内容 | 说明 |
|------|------|------|
| 1 | 数据结构变更 | 新增字段、修改字段、新建数据类 |
| 2 | 核心逻辑流程 | 用文字或伪代码描述关键处理流程 |
| 3 | 入口与出口 | 从哪里触发，到哪里结束 |
| 4 | 影响范围 | 哪些现有模块可能受影响 |
| 5 | 边界场景 | 异常、空值、网络失败、并发等 |

> 文件路径的定位和修改点拆解留给 task-spec-generator，本阶段不做。

> 先说改哪里，再说怎么改，最后说注意什么。

**全局约束标记**：如果该任务受全局约束影响，在「数据结构变更」之前插入 `**⚠️ 全局约束：** 描述内容`。

**如果任务涉及多个模块：**
- 按模块分别列出数据结构和设计
- 用架构图或文字描述模块间的调用关系

---

按下方模板生成 `{OUT}/tech-design.md`：

## T-001 — 登录接口接入

**关联任务：** T-001
**关联需求：** REQ-001

### 数据结构变更

**LoginResponse（新增）：**
- `token: string` — 鉴权令牌
- `userId: string` — 用户 ID
- `expiresIn: number` — 过期时间（秒）

**用户信息模型（修改）：**
- 新增 `lastLoginTime: string` 字段

### 核心逻辑流程

1. 用户输入手机号和验证码 → 调用登录页校验逻辑
2. 校验通过 → 调用 `LoginApi.sendCode(phone)` 发送验证码
3. 收到验证码 → 调用 `LoginApi.login(phone, code)`
4. 登录成功 → 保存 token 到本地存储，跳转到首页
5. 登录失败 → 根据错误码提示用户

### 入口与出口

- **入口：** 登录页点击「登录」按钮
- **出口：** 跳转至首页 / 停留在登录页提示错误

### 影响范围

- 影响认证模块的 token 刷新逻辑
- 影响首页的用户信息展示

### 边界场景

- 验证码倒计时期间重复点击 → 按钮置灰
- 网络超时 → 显示重试提示
- Token 过期 → 触发静默刷新或跳转登录页

## T-003 — 头像上传 [G]

**关联任务：** T-003
**关联需求：** REQ-002

**⚠️ 全局约束：** 上传能力需覆盖应用内所有需要头像的场景

### 数据结构变更

无新增模型，复用现有 `UploadResponse`。

### 核心逻辑流程

1. 用户点击头像 → 弹出选择（拍照 / 相册）
2. 选择图片 → 校验大小和格式
3. 校验通过 → 调用 `FileUploader.upload(file)`
4. 上传成功 → 更新头像 URL，刷新 UI
5. 上传失败 → 重试或提示

### 入口与出口

- **入口：** 个人主页点击头像
- **出口：** 头像更新完成 / 上传失败提示

### 影响范围

- 影响个人主页、设置页等所有展示头像的位置

### 边界场景

- 图片超过 5MB → 拒绝并提示
- 格式非 jpg/png → 拒绝并提示
- 上传中断（网络断开）→ 支持断点续传

## 修订模式

### 模式判断

检查 `{OUT}/tech-design.md` 的文档状态：
- 状态为「已确认」→ 告知用户：**「技术方案已完成（当前 vX.X），无需修订」**，终止
- 状态为「待审核」→ 进入修订流程

### 处理审核

**待澄清条目：**
- 扫描每个 CL 详情块中的「人工决策」行
- 有内容 → AI 复核：
  - 已明确解决 → 按决策调整方案，CL 状态改「已解决」
  - 不清晰或不完整 → CL 状态改「需补充」，AI 复核附说明
- 人工决策为空 → 保持「待处理」

用户消息中的额外审核意见一并处理。

### 追加修订记录

在「修订记录」表格中追加一行。版本号读取现有记录最后一行递增，无记录时从 v1.0 开始。

### 检查完成标准

所有 CL 条目的「状态」均为「已解决」→ 文档状态改为「已确认」。
任一项未满足 → 保持「待审核」。
