# 代码生成

按开发规格逐修改点生成代码，写入项目代码仓库，产出 `output/report-T-XXX.md`。

> 在 workspace 目录下执行。

## 前置检查

当前目录存在 `CONTEXT.md`。不存在 → 报错终止。

## 执行流程

### Step 1：确认目标

确定要实现的规格文件：

- 用户指定 T-XXX → 直接使用
- 未指定 → 读 CONTEXT.md 规格索引，按以下规则选择：
  1. 排除已标记 ✅ 的任务
  2. 优先选无依赖或依赖已 ✅ 的任务
  3. 同优先级按 T-xxx 编号顺序

无可用任务 → 告知用户：**「所有任务已完成」**。

`output/specs/T-XXX.md` 存在。不存在 → 报错：**「未找到 specs/T-XXX.md」**，终止。

### Step 2：读取规格

读 `output/specs/T-XXX.md`，理解：任务概述、依赖关系、关键行为、实现方案、修改点清单。

检查依赖：从 spec 的「依赖」段提取，对照 CONTEXT.md 规格索引。如果用户手动指定了有未完成依赖的 T-XXX → 警告用户但继续执行。

读 `CONTEXT.md`，提取代码路径。代码路径为空 → 报错：**「CONTEXT.md 中未配置代码路径，无法生成代码」**。

### Step 3：生成代码

对每个修改点，**以实现任务为目标，不盲从 spec**：

1. **定位文件**：规格中的文件路径，`[NEW]` 标记的创建新文件（父目录不存在则自动创建）
2. **读取现有代码**：修改已有文件时，先读取理解上下文
3. **生成修改**：按规格指令和约束条件生成代码
4. **不超出规格范围**：只修改相关文件和位置，不做额外"优化"

过程中逐修改点追踪：记录实际修改的文件、类型、是否与 spec 一致。一致标 ✅，不一致标 ⚠️。

**遇到以下情况时写入 ISSUES.md，不自行决定：**

- spec 与实际代码结构不符（文件不存在、类名错误等）
- 规格描述的实现方式无法落地或有明显缺陷
- 发现有更好的实现方案
- 关键行为描述与修改点指令矛盾

> 偏差只记录在 report 中，**不修改 spec**。偏差合理时，由人工根据 report 让 Agent 统一回写 spec 和 design。

### Step 4：生成报告

`output/` 目录不存在则先创建。生成 `output/report-T-XXX.md`：

```markdown
# 代码生成报告 — T-XXX [任务标题]

> 生成日期：YYYY-MM-DD · 来源：specs/T-XXX.md

## 修改清单

| # | 文件 | 类型 | 按规格 | 说明 |
|---|------|------|--------|------|
| 1 | `src/api/auth.ets` | 新增 | ✅ | 按 spec 修改点 1 |
| 2 | `src/services/AuthManager.ets` | 修改 | ⚠️ 偏离 | 代码库已有 AuthManager，放入 AuthService 会职责重复 |

## 偏离说明

> 无偏离则写「无」。

### 偏离 1 — [简述]

- **spec 要求：** [spec 中的原始描述]
- **实际实现：** [做了什么、为什么不同]
- **原因：** [偏离的原因]
- **影响：** [可能影响的产物，如 design.md 某节]
```

### Step 5：更新工作空间文件

**更新 CONTEXT.md：**

- 在 `规格` 索引中标注已完成的任务（`T-001 — ✅ [report](output/report-T-001.md)`）
- 在 `## 代码产出` 表格中标注（`T-001 | ✅ 已完成`）
- 如全部任务完成 → `阶段` 改为 `代码生成完成`，`下一步` 改为 `执行 wf-test-generator`

**写入 ISSUES.md（Step 3 中发现的问题）：**

在 `## 实现阶段` 下按以下格式追加，编号按已有 Q-xxx 递增：

```
### Q-001 — [问题简述]

- **问题：** [具体描述]
- **AI 建议：** [建议方案]
- **影响：** [影响的文件或任务]
- **提出：** YYYY-MM-DD wf-code-generator

---

**人工决策：**
**状态：** 待决策
```

**写入 JOURNAL.md：**

在文件末尾追加当天日期标题（如已存在则复用），然后追加：

```
### HH:MM — wf-code-generator 完成 T-XXX

- 任务：[标题]
- 修改 X 个文件，其中 X 处偏离 spec
- 生成 output/report-T-XXX.md
- 发现问题 X 项，写入 ISSUES.md
```

## 输出前自检

- [ ] 每个修改点已生成对应代码
- [ ] 代码遵循规格中的关键行为和约束条件
- [ ] 未修改规格列出的文件之外的代码
- [ ] `output/report-T-XXX.md` 已生成，偏离有原因和影响说明
- [ ] report 中有 ⚠️ 的偏离均已写入 ISSUES.md（需人工决策的场景）
