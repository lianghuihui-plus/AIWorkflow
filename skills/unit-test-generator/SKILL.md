# 单元测试用例生成

从 `generated/T-XXX-report.md` 读取代码生成报告，结合规格文档和实际代码，逐修改点生成单元测试用例，写入项目代码仓库，并在 `tests/` 目录下输出测试生成报告。

> 本阶段做**测试落地**：代码已确认后，为每个修改点生成对应的单元测试。不运行、不执行，这些留给后续阶段。每次执行处理一个 spec。

## 配置读取

### 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

定位到 `workflow.yaml` 后，立即解析并执行：

**除工作流目录定位外，本阶段所有输入从 `workflow.yaml` 读取，不交互询问。**

### 读取并校验

1. 解析 `workflow.yaml`，校验 YAML 格式
2. 校验必填字段：
   - `project.code_path` — 如为空 → 终止并告知用户：**「workflow.yaml 中缺少 project.code_path，无法写入测试」**
   - `project.platform` — 如为空 → 终止并告知用户
3. 提取可选字段：
   - `project.name` — 用于文档标题
4. 提取 `output.base_dir`（默认 `output`），记为 `OUT`
5. 记 `CODE_PATH` = `project.code_path`，`PLATFORM` = `project.platform`

YAML 格式非法导致无法解析 → 终止并告知用户具体错误。后续步骤直接使用上述变量，不再重复读取。

### 配置字段说明

| 配置路径 | 含义 | 必填 |
|---------|------|------|
| `project.code_path` | 项目代码路径 | 是 |
| `project.platform` | 目标平台 | 是 |
| `output.base_dir` | 输出根目录 | 是（默认 `output`） |
| `project.name` | 项目名称 | 否 |

### 读取测试配置

读取 `AGENT.md`（由 dev-profile 加载）中的「单元测试」段落，作为测试生成的优先约束。段落内容由用户按项目实际情况填写，AI 直接按其中指示执行。

未配置或 AGENT.md 不存在 → 根据 `PLATFORM` 和项目依赖自动检测测试框架与目录。

## 编号规则

| 编号 | 格式 | 说明 |
|------|------|------|
| 任务 ID | T-001、T-002… | 保持 specs 中的编号不变 |
| 修改点 ID | M-001、M-002… | 保持 specs 中的编号不变 |
| 待澄清 ID | CL-001、CL-002… | 按检出顺序分配 |

## 工作流程

输出：
- 测试文件写入 `CODE_PATH` 下的对应测试目录
- 测试生成报告：`{OUT}/tests/T-XXX-test-report.md`

### 前置检查

检查 `{OUT}/specs/README.md`：
- 不存在 → 终止并告知用户：**「未找到 specs/README.md，请先执行 task-spec-generator」**

读取 `{OUT}/specs/README.md`，获取所有任务列表。对每个任务，检查：
1. `{OUT}/generated/T-XXX-report.md` 是否存在及状态
2. `{OUT}/tests/T-XXX-test-report.md` 是否存在及状态

如 `{OUT}/generated/` 目录不存在或其中没有任何「已审核」状态的报告 → 终止并告知用户：**「未找到已审核的代码生成报告，请先执行 code-generator 并完成代码审核」**

> 任何步骤遇到不可恢复的错误，终止并将错误信息告知用户，不静默跳过。

### 任务选择

**1. 用户已指定 → 直接使用**

用户消息中提到了具体的任务 ID（如 `T-001`），直接以该任务为目标，跳过选择流程。

**2. 用户未指定 → 自动定位首个待处理的任务**

在 spec 列表中按顺序找到第一个满足以下条件的任务：
- `generated/T-XXX-report.md` 存在且状态为「已审核」（代码已确认）
- `tests/T-XXX-test-report.md` 不存在，或状态为「待审核」（测试未完成）

找到后向用户确认：

> 检测到首个待生成测试的任务为 **T-XXX — [标题]**（[N] 个修改点），是否生成测试？

用户确认后继续；用户拒绝则终止，由用户自行指定。

如果所有任务的测试报告状态均为「已审核」→ 告知用户：**「所有任务测试已完成」**，终止。

---

**选定后的校验规则（两种方式均适用）：**

- `{OUT}/specs/T-XXX-spec.md` 不存在 → 终止并告知：**「未找到 T-XXX-spec.md，请先执行 task-spec-generator」**
- `{OUT}/generated/T-XXX-report.md` 不存在 → 终止并告知：**「未找到 T-XXX 的代码生成报告，请先执行 code-generator」**
- `{OUT}/generated/T-XXX-report.md` 状态不是「已审核」→ 终止并告知用户：**「T-XXX 代码尚未审核通过（当前状态：xxx），请先完成代码审核」**
- `{OUT}/tests/T-XXX-test-report.md` 不存在 → 进入「生成流程」
- `{OUT}/tests/T-XXX-test-report.md` 存在且状态为「待审核」→ 跳过生成流程，直接进入「修订模式」
- `{OUT}/tests/T-XXX-test-report.md` 存在且状态为「已审核」→ 告知并询问是否覆盖重新生成

---

### 生成流程

#### Step 1：收集上下文

1. 读取 `{OUT}/specs/T-XXX-spec.md` 全文，提取修改点清单（M-001、M-002…）及每个修改点的详细信息
2. 读取 `{OUT}/generated/T-XXX-report.md`，提取「修改点执行结果」表，获取：
   - 每个 M-XXX 的操作类型（新增 / 修改 / 删除）
   - 目标文件路径
   - 执行结果（✅ / ⚠️ / ❌）
3. 对执行结果为 ✅ 或 ⚠️ 的修改点，读取 `{CODE_PATH}/{目标文件路径}` 获取实际代码
4. 读取测试配置（AGENT.md 或自动检测结果）

#### Step 2：逐修改点生成测试

按 M-001、M-002… 顺序遍历每个修改点，按操作类型分别处理。

**纯删除修改点：**

执行结果为 ✅ 且操作类型为「删除」→ 无需生成测试，在报告中标注「无需测试」。

**新增文件 / 修改文件：**

1. 根据检测到的测试配置（AGENT.md 或自动检测结果）确定测试文件的路径和命名
2. 检查测试文件是否已存在：
   - 不存在 → 新建测试文件
   - 已存在 → 在现有测试文件中追加用例，保持原有用例不变
3. 分析目标代码文件，识别可测试单元：
   - 函数/方法 → 生成输入输出验证用例
   - 数据模型 → 生成构造和字段验证用例
   - 组件 → 生成渲染和交互用例
   - 异步逻辑 → 生成成功和失败分支用例
4. 为每个可测试单元生成对应测试用例，包含：
   - 正常路径（happy path）
   - 边界值（空值、极值、特殊字符）
   - 异常路径（错误输入、网络失败等）
5. 将测试代码写入测试文件

**生成原则：**
- 测试框架和风格严格遵循检测到的测试框架
- 如果项目已有测试文件，匹配已有测试文件的风格（命名、断言方式、结构）
- 不确定如何测试的依赖（如网络层、数据库）→ 生成测试框架，用注释标注依赖点，记录到报告的「待澄清」表中
- 生成有意义的断言，不使用 `expect(true).assertTrue()` 等占位断言

**失败处理：**

遇到以下无法继续的情况时，终止本次执行并反馈用户：
- 目标测试目录不存在且无法创建（权限不足）
- 磁盘空间不足导致写入失败

已成功写入的修改点保留不做回滚。

#### Step 3：输出测试报告

**生成规则：**
- 测试覆盖总览表中，结果列：✅ 成功 / ⚠️ 部分覆盖（有待澄清） / — 无需测试
- 用例数为实际生成的测试用例数量
- 待澄清 ID 按检出顺序分配 CL-001、CL-002…
- 报告状态初始为「待审核」

写入 `{OUT}/tests/T-XXX-test-report.md`：

```markdown
# T-XXX 测试生成报告 — [任务标题]

> 生成日期：YYYY-MM-DD
> 来源：specs/T-XXX-spec.md, generated/T-XXX-report.md
> 平台：[platform]
> 测试框架：[detected framework]
> 状态：待审核

---
```

### 修订模式

#### Step 1：模式判断

检查 `{OUT}/tests/T-XXX-test-report.md` 的文档状态：
- 状态为「已审核」→ 告知用户：**「测试报告已审核通过（当前 vX.X），如需修改请直接调整测试文件」**，终止
- 状态为「待审核」→ 继续处理
- 状态为其他值或无法解析 → 视为「待审核」，继续处理，并在操作提示中标注异常状态值

#### Step 2：收集修订依据

扫描报告中的「待澄清」章节，汇总所有未解决的 CL 条目（状态为「待处理」）。

同时检查用户消息中是否包含额外的修改意见。

如待澄清表全部已解决且用户未提供修改意见 → 告知用户：**「当前没有可修订的内容，待澄清项均已解决。如确认测试无误，请明确告知审核通过」**，终止。

#### Step 3：执行修订

根据用户填写的「人工决策」或用户消息中的修改意见，逐条处理：
1. 按决策修改对应的测试文件
2. 在报告的「修订记录」表格末尾追加一条修订记录（新条目始终追加在表格最后一行之后，不在顶部插入）
3. 更新对应 CL 详情块：填写「AI 复核」、「状态」改为「已解决」；如决策不清晰，保持「待处理」，AI 复核附说明（告知用户哪些点仍需澄清）。**⚠️ 已解决的 CL 条目和详情块必须保留，不得删除。**

扫描用户消息中提到的 CL ID 或修改点 ID，将对应审核意见应用到匹配条目；无明确 ID 时按关键词匹配。已解决条目不再处理，除非审核人明确要求。

#### Step 4：收敛

所有 CL 条目均为「已解决」，且用户**明确告知测试审核通过**（如「审核 OK」「没问题」等）→ 报告状态改为「已审核」。

仅有 CL 全部解决但用户未明确确认 → 保持「待审核」，提示用户：**「待澄清项已全部解决，请确认测试审核结果」**。

> 修订完成后执行「输出前自检」。

## 输出前自检

生成或修订后，确认以下全部通过：

- [ ] 测试文件已写入 `CODE_PATH` 下正确的测试目录
- [ ] 测试框架和风格与项目已有测试一致
- [ ] 测试覆盖总览表中每个有代码变更的 M-ID 都有对应行
- [ ] 待澄清总览表与详情块一一对应，无孤立的条目
- [ ] 跳过的修改点均有明确的跳过原因记录（如「纯删除，无需测试」）
- [ ] 修订记录版本号连续，最新一行在最下
- [ ] 文档状态与实际完成标准一致
