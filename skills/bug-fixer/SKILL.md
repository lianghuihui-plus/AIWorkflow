# Bug 修复

读取 `{OUT}/bugs/BUG-XXX.md`，分析代码、确定修复方案、执行修复，回写 bug 文档。

> 本 skill 只处理**单个** bug 文档。不批量修复。

## 配置读取

### 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

定位到 `workflow.yaml` 后，立即解析并执行。

### 读取并校验

1. 解析 `workflow.yaml`，校验 YAML 格式
2. 校验必填字段：
   - `project.code_path` — 如为空 → 终止并告知用户：**「workflow.yaml 中缺少 project.code_path，无法写入代码」**
3. 提取 `output.base_dir`（默认 `output`），记为 `OUT`
4. 记 `CODE_PATH` = `project.code_path`

YAML 格式非法导致无法解析 → 终止并告知用户具体错误。

### 配置字段说明

| 配置路径 | 含义 | 必填 |
|---------|------|------|
| `project.code_path` | 项目代码路径 | 是 |
| `output.base_dir` | 输出根目录 | 是（默认 `output`） |

## 工作流程

### 前置检查

检查 `{OUT}/bugs/` 目录：
- 不存在或为空 → 终止并告知用户：**「未找到 bug 文档，请先执行 bug-new 生成」**

扫描目录下所有 `BUG-XXX.md`，统计各状态数量。

如所有 bug 文档状态均为「已修复」→ 告知用户：**「所有 bug 已修复」**，终止。

### Bug 选择

**1. 用户已指定 → 直接使用**

用户消息中提到了具体的 BUG ID（如 `BUG-001`），直接以该 bug 为目标。

**2. 用户未指定 → 自动定位首个待修复的 bug**

扫描 `{OUT}/bugs/` 目录，按编号顺序找到第一个状态为「待修复」的 bug 文档，向用户确认：

> 检测到待修复 bug：**BUG-XXX — [问题简述]**，是否修复？

用户确认后继续；用户拒绝则终止，由用户自行指定。

---

**选定后校验：**

- `{OUT}/bugs/BUG-XXX.md` 不存在 → 终止并告知
- 状态为「待修复」→ 进入修复流程
- 状态为「已修复」→ 告知用户：**「BUG-XXX 已修复（当前状态：已修复），如需重新修复请将状态手动改为待修复」**，终止
- 状态无法解析 → 视为「待修复」，标注异常

---

### 修复流程

#### Step 1：读取 bug 文档

读取 `{OUT}/bugs/BUG-XXX.md`，提取：
- 问题描述 — 了解 bug 现象
- 影响范围 — 获取涉及的文件列表
- AI 初步判断 — 参考修复方向

#### Step 2：分析代码

对影响范围的每个文件，读取 `{CODE_PATH}/{文件路径}` 的实际代码，理解当前实现。

#### Step 3：确定修复方案

根据问题描述和代码分析，确定：
- **原因** — bug 的根因
- **修复方案** — 具体的修改步骤
- **改动范围** — 实际需要改哪些文件（可能比影响范围多或少）
- **验证方式** — 如何验证修复有效（手动检查点 / 测试用例 / 兼有）

#### Step 4：执行修复

按修复方案逐文件修改：
1. 修改代码文件 — 遵循项目现有风格，仅改动问题相关部分
2. 同步更新测试文件 — 修复导致的测试变更（新增用例、调整断言等）
3. 代码中关键修复处加简要注释

**修复原则：**
- 最小改动原则，不改无关代码
- 匹配项目现有代码风格
- 如果项目已有测试文件，匹配已有测试风格

#### Step 5：回写 bug 文档

1. 填写「Bug 修复」段落（原因、修复方案、改动范围、验证）
2. 在「修订记录」末尾追加一行（版本递增，操作列填 `bug-fixer`）
3. 状态改为「已修复」

## 输出前自检

修复后确认：

- [ ] 代码文件已修改并写入 `CODE_PATH`
- [ ] 测试文件已同步更新
- [ ] bug 文档「Bug 修复」段已填写完整
- [ ] bug 文档状态已改为「已修复」
- [ ] 修订记录已追加，版本号连续
