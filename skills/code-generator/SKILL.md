# 代码生成

从 `specs/` 目录读取单个任务的开发规格文档，逐条执行修改点，在项目代码仓库中直接生成/修改代码。

> 本阶段做**代码落地**：规格文件定义改什么，本阶段直接改项目代码。每次执行处理一个 spec。不运行、不测试，这些留给后续阶段。

## 配置读取

### 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

定位到 `workflow.yaml` 后，立即解析并执行：

**除工作流目录定位外，本阶段所有输入从 `workflow.yaml` 读取，不交互询问。**

### 读取并校验

1. 解析 `workflow.yaml`，校验 YAML 格式
2. 校验必填字段：
   - `project.code_path` — 如为空 → 终止并告知用户：**「workflow.yaml 中缺少 project.code_path，无法写入代码」**
   - `project.platform` — 如为空 → 终止并告知用户
3. 提取可选字段：
   - `project.name` — 用于文档标题
4. 提取 `output.base_dir`（默认 `output`），记为 `OUT`
5. 记 `CODE_PATH` = `project.code_path`，`PLATFORM` = `project.platform`

YAML 格式非法导致无法解析 → 终止并告知用户具体错误。后续步骤直接使用上述变量，不再重复读取。

### 配置字段说明

| 配置路径 | 含义 | 必填 |
|---------|------|------|
| `project.code_path` | 项目代码路径 | 是 |
| `project.platform` | 目标平台 | 是 |
| `output.base_dir` | 输出根目录 | 是（默认 `output`） |
| `project.name` | 项目名称 | 否 |

## 代码生成规范

### 根据平台适配

生成的代码必须符合目标平台的语法和命名规范：

**HarmonyOS (ArkTS)：**
- 导入路径使用 `import { X } from '...'` 格式
- 类/组件导出使用 `export class / export struct`
- 网络请求使用 `@ohos.net.http`
- 遵循 `features/{模块}/src/main/ets/` 目录结构

**Android (Kotlin)：**
- 遵循 `{模块}/src/main/java/{包名}/` 目录结构
- 导入使用 `import {包名}.{类名}` 格式

### 代码风格

- 符号命名：跟随项目现有规范（驼峰、下划线等）
- 注释：关键逻辑处简要说明，复杂逻辑用多行注释
- 缩进：跟随项目现有风格
- 不引入项目未使用的第三方库

## 工作流程

输出：
- 代码写入 `CODE_PATH` 下的项目文件
- 生成报告：`{OUT}/generated/T-XXX-report.md`

### 前置检查

检查 `{OUT}/specs/README.md`：
- 不存在 → 终止并告知用户：**「未找到 specs/README.md，请先执行 task-spec-generator」**

读取 `{OUT}/specs/README.md`，获取所有任务列表。对每个任务，检查 `{OUT}/generated/T-XXX-report.md` 是否存在及状态，确定生成状态。

> 任何步骤遇到不可恢复的错误，终止并将错误信息告知用户，不静默跳过。

### 任务选择

**1. 用户已指定 → 直接使用**

用户消息中提到了具体的任务 ID（如 `T-001`），直接以该任务为目标，跳过选择流程。

**2. 用户未指定 → 自动定位首个未完成的任务**

在 spec 列表中按顺序找到第一个**代码报告未审核完成**的任务（报告不存在，或报告状态为「待审核」），向用户确认：

> 检测到首个待处理的任务为 **T-XXX — [标题]**（[N] 个修改点），是否处理？

用户确认后继续；用户拒绝则终止，由用户自行指定。

如果所有任务的报告状态均为「已审核」→ 告知用户：**「所有任务代码已审核完成」**，终止。

---

**选定后的校验规则（两种方式均适用）：**

- `{OUT}/specs/T-XXX-spec.md` 不存在 → 终止并告知：**「未找到 T-XXX-spec.md，请先执行 task-spec-generator」**
- spec 文档状态不是「已确认」→ 终止并告知用户：**「T-XXX 规格尚未确认（当前状态：xxx），请先完成审核收敛」**
- `{OUT}/generated/T-XXX-report.md` 不存在 → 进入「生成流程」
- `{OUT}/generated/T-XXX-report.md` 存在且状态为「待审核」→ 跳过生成流程，直接进入「修订模式」
- `{OUT}/generated/T-XXX-report.md` 存在且状态为「已审核」→ 告知并询问是否覆盖重新生成

---

### 生成流程

#### Step 1：读取规格

读取 `{OUT}/specs/T-XXX-spec.md` 全文，提取修改点清单（M-001、M-002…）及每个修改点的详细信息。

#### Step 2：逐修改点生成代码

按 M-001、M-002… 顺序遍历每个修改点，按操作类型分别处理。

**新增文件：**

1. 根据规格中的「修改内容」生成完整文件内容
2. 文件内容必须包含：
   - 必要的 `import` 语句（参考项目现有文件的导入规范）
   - 类/组件/函数定义
   - 方法实现（按规格中的处理逻辑逐条实现）
3. 处理该修改点关联的边界场景（见下方「边界场景处理」）
4. 写入 `{CODE_PATH}/{文件路径}`

生成原则：
- 参考项目现有代码的命名规范、目录结构、导入路径习惯
- 方法签名严格按规格中的签名生成
- 处理逻辑逐条翻译为代码，不遗漏
- 规格中标注了「需确认」的修改点 → 跳过，记录到报告的「待澄清」表中

**修改文件：**

1. 读取 `{CODE_PATH}/{文件路径}` 获取现有文件内容
2. 根据规格中的修改内容，在现有文件基础上直接修改
3. 处理该修改点关联的边界场景
4. 写回 `{CODE_PATH}/{文件路径}`

生成原则：
- 基于原文件修改，保留不相关的代码不变
- 仅改动规格中描述的部分
- 新增代码与原代码风格一致
- 不确定如何实现的部分 → 跳过，记录到报告的「待澄清」表中

**删除文件：**

直接删除 `{CODE_PATH}/{文件路径}`。

**边界场景处理：**

对当前修改点关联的边界场景，在生成代码时同步覆盖：
- 已覆盖 → 在代码中加注释标注对应场景
- 未覆盖 → 补充对应逻辑
- 无法确定如何覆盖 → 跳过，记录到报告的「待澄清」表中

**失败处理：**

遇到以下无法继续的情况时，终止本次执行并反馈用户：
- 目标目录不存在或无写入权限
- 磁盘空间不足导致写入失败

已成功写入的修改点保留不做回滚。

#### Step 3：输出生成报告

**生成规则：**
- 修改点执行结果表中，结果列：✅ 成功 / ⚠️ 跳过 / ❌ 失败
- 修改点执行结果表中，备注列记录跳过或失败原因
- 待澄清 ID 按检出顺序分配 CL-001、CL-002…
- 报告状态初始为「待审核」

写入 `{OUT}/generated/T-XXX-report.md`：

```markdown
# T-XXX 代码生成报告 — [任务标题]

> 生成日期：YYYY-MM-DD
> 来源：specs/T-XXX-spec.md
> 平台：[platform]
> 状态：待审核
```

### 修订模式

#### Step 1：模式判断

检查 `{OUT}/generated/T-XXX-report.md` 的文档状态：
- 状态为「已审核」→ 告知用户：**「代码生成报告已审核通过（当前 vX.X），如需修改请直接调整代码」**，终止
- 状态为「待审核」→ 继续处理
- 状态为其他值或无法解析 → 视为「待审核」，继续处理，并在操作提示中标注异常状态值

#### Step 2：收集修订依据

扫描报告中的「待澄清」章节，汇总所有未解决的 CL 条目（状态为「待处理」）。

同时检查用户消息中是否包含额外的修改意见。

如待澄清表全部已解决且用户未提供修改意见 → 告知用户：**「当前没有可修订的内容，待澄清项均已解决。如确认代码无误，请明确告知审核通过」**，终止。

#### Step 3：执行修订

根据用户填写的「人工决策」或用户消息中的修改意见，逐条处理：
1. 按决策修改对应的项目代码文件
2. 在报告的「修订记录」表格末尾追加一条修订记录（新条目始终追加在表格最后一行之后，不在顶部插入）
3. 更新对应 CL 详情块：填写「AI 复核」、「状态」改为「已解决」；如决策不清晰，保持「待处理」，AI 复核附说明（告知用户哪些点仍需澄清）。**⚠️ 已解决的 CL 条目和详情块必须保留，不得删除。**

扫描用户消息中提到的 CL ID 或修改点 ID，将对应审核意见应用到匹配条目；无明确 ID 时按关键词匹配。已解决条目不再处理，除非审核人明确要求。

#### Step 4：收敛

所有 CL 条目均为「已解决」，且用户**明确告知代码审核通过**（如「审核 OK」「没问题」等）→ 报告状态改为「已审核」。

仅有 CL 全部解决但用户未明确确认 → 保持「待审核」，提示用户：**「待澄清项已全部解决，请确认代码审核结果」**。

> 修订完成后执行「输出前自检」。

## 输出前自检

生成或修订后，确认以下全部通过：

- [ ] 代码已写入正确的 `CODE_PATH` 目录下
- [ ] 代码风格与项目现有代码一致
- [ ] 修改点执行结果表中每个 M-ID 都有对应行
- [ ] 待澄清总览表与详情块一一对应，无孤立的条目
- [ ] 跳过的修改点均有明确的跳过原因记录
- [ ] 修订记录版本号连续，最新一行在最下
- [ ] 文档状态与实际完成标准一致
