# 代码生成

从 `specs/` 目录读取单个任务的开发规格文档，逐条执行修改点，在项目代码仓库中直接生成/修改代码。

> 本阶段做**代码落地**：规格文件定义改什么，本阶段直接改项目代码。每次执行处理一个 spec。不运行、不测试，这些留给后续阶段。

## 输入与输出

## Step 1 — 定位工作流目录

按以下优先级定位 `workflow.yaml` 所在的工作流根目录：

1. **检查当前目录** — 当前工作目录中存在 `workflow.yaml` → 从该目录读取配置
2. **检查会话上下文** — 如本会话中已运行过 `workflow-init` 或其他阶段 skill，且已知工作流目录路径 → 使用该路径
3. **询问用户** — 以上均未找到时，向用户询问：
   > 我需要知道当前工作流目录路径，请提供 workflow-xxx 目录的路径。

验证定位到的目录中存在 `workflow.yaml`；不存在则提示用户先执行 `workflow-init` 初始化。

**本阶段除 Step 1 外所有输入从 `workflow.yaml` 读取，不交互询问。**

### 输入（workflow.yaml）

| 配置路径 | 用于 | 说明 |
|---------|------|------|
| `output.base_dir` | 输出根目录 | 必填。默认 `output` |
| `project.platform` | 目标平台 | 必填，如 HarmonyOS / Android / iOS |
| `project.code_path` | 项目代码路径 | 必填。代码直接写入该路径下的项目文件 |
| `project.name` | 项目名称 | 用于文档标题 |

### 输入（前序阶段输出）

| 文件 | 用于 | 说明 |
|------|------|------|
| `{output.base_dir}/specs/T-XXX-spec.md` | 当前任务的修改点规格 | 逐条生成代码 |

### 输出

- **代码文件**：直接写入 `{code_path}/` 下的项目文件中（新增、修改、删除）
- **生成报告**：写入 `{output.base_dir}/generated/T-XXX-report.md`

## 工作流程

### Step 2：读取配置并选择任务

读取 `workflow.yaml` 获取配置，记为：
- `OUT`：`output.base_dir`（默认 `output`）
- `CODE_PATH`：`project.code_path`（相对于 workflow-xxx 根目录）
- `PLATFORM`：`project.platform`

读取 `{OUT}/specs/README.md`，获取所有任务。对每个任务，检查 `{OUT}/generated/T-XXX-report.md` 是否存在且文档状态，确定生成状态。

向用户展示任务列表，让用户选择本次要处理的任务：

```
以下任务可供生成代码，请选择：

| 编号 | 任务 ID | 标题 | 修改点数 | 状态 |
|------|--------|------|---------|------|
| 1 | T-001 | 登录接口接入 | 3 | ⬜ 未生成 |
| 2 | T-002 | 登录页交互 | 2 | ⬜ 待审核 |
| 3 | T-003 | 头像上传 | 1 | ✅ 已审核 |

请输入要处理的任务编号或任务 ID：
```

交互与校验规则：
- 用户可直接回复任务编号（如 `1`）、任务 ID（如 `T-001`）
- 只接收一个任务，不批量处理
- 所有任务均已生成（状态为「待审核」或「已审核」）→ 告知用户：**「所有任务代码已生成完毕」**，终止
- `{OUT}/specs/T-XXX-spec.md` 不存在 → 终止并告知：**「未找到 T-XXX-spec.md，请先执行 task-spec-generator」**
- 用户选择的 spec 文档状态不是「已确认」→ 提示：**「T-XXX 规格尚未确认（当前状态：xxx），是否仍要生成？」**，等待确认
- `{OUT}/generated/T-XXX-report.md` 不存在 → 进入「生成流程」
- `{OUT}/generated/T-XXX-report.md` 存在且状态为「待审核」→ 跳过生成流程，直接进入「修订模式」
- `{OUT}/generated/T-XXX-report.md` 存在且状态为「已审核」→ 告知并询问是否覆盖重新生成

## 生成流程

### Step 3：逐修改点生成代码

按 M-001、M-002… 顺序遍历 spec 文件中的每个修改点，按操作类型分别处理。

#### 新增文件

规格中「操作」为「新增文件」时：

1. 根据规格中的「修改内容」生成完整文件内容
2. 文件内容必须包含：
   - 必要的 `import` 语句（参考项目现有文件的导入规范）
   - 类/组件/函数定义
   - 方法实现（按规格中的处理逻辑逐条实现）
3. 处理该修改点关联的边界场景（见下方「边界场景处理」）
4. 写入 `{CODE_PATH}/{文件路径}`

**生成原则：**
- 参考项目现有代码的命名规范、目录结构、导入路径习惯
- 方法签名严格按规格中的签名生成
- 处理逻辑逐条翻译为代码，不遗漏
- 规格中标注了「需确认」的修改点 → 跳过，记录到报告的「待澄清」表中

#### 修改文件

规格中「操作」为「修改文件」时：

1. 读取 `{CODE_PATH}/{文件路径}` 获取现有文件内容
2. 根据规格中的修改内容，在现有文件基础上直接修改
3. 处理该修改点关联的边界场景（见下方）
4. 写回 `{CODE_PATH}/{文件路径}`

**生成原则：**
- 基于原文件修改，保留不相关的代码不变
- 仅改动规格中描述的部分
- 新增代码与原代码风格一致
- 不确定如何实现的部分 → 跳过，记录到报告的「待澄清」表中

#### 删除文件

规格中「操作」为「删除文件」时：直接删除 `{CODE_PATH}/{文件路径}`。

#### 边界场景处理

对当前修改点关联的边界场景，在生成代码时同步覆盖：
- 已覆盖 → 在代码中加注释标注对应场景
- 未覆盖 → 补充对应逻辑
- 无法确定如何覆盖 → 跳过，记录到报告的「待澄清」表中

#### 失败处理

遇到以下无法继续的情况时，终止本次执行并反馈用户：
- 目标目录不存在或无写入权限
- 磁盘空间不足导致写入失败

已成功写入的修改点保留不做回滚。

## 代码生成规范

### 根据平台适配

生成的代码必须符合目标平台的语法和命名规范：

**HarmonyOS (ArkTS)：**
- 导入路径使用 `import { X } from '...'` 格式
- 类/组件导出使用 `export class / export struct`
- 网络请求使用 `@ohos.net.http`
- 遵循 `features/{模块}/src/main/ets/` 目录结构

**Android (Kotlin)：**
- 遵循 `{模块}/src/main/java/{包名}/` 目录结构
- 导入使用 `import {包名}.{类名}` 格式

### 代码风格

- 符号命名：跟随项目现有规范（驼峰、下划线等）
- 注释：关键逻辑处简要说明，复杂逻辑用多行注释
- 缩进：跟随项目现有风格
- 不引入项目未使用的第三方库

## 修订模式

### 模式判断

检查 `{OUT}/generated/T-XXX-report.md` 的文档状态：
- 状态为「已审核」→ 告知用户：**「代码生成报告已审核通过（当前 vX.X），如需修改请直接调整代码」**，终止
- 状态为「待审核」→ 进入修订流程

### 收集修订依据

扫描报告中的「待澄清」章节，汇总所有未解决的 CL 条目（状态为「待处理」或「需补充」），整理为修订清单展示给用户。

同时检查用户消息中是否包含额外的修改意见。

如待澄清表全部已解决且用户未提供修改意见 → 告知用户：**「当前没有可修订的内容，待澄清项均已解决。如确认代码无误，请明确告知审核通过」**，终止。

### 执行修订

根据用户填写的「人工决策」或用户消息中的修改意见，逐条处理：
1. 按决策修改对应的项目代码文件
2. 在报告的「修订记录」中追加一条修订记录
3. 更新对应 CL 详情块：填写「AI 复核」、「状态」改为「已解决」；如决策不清晰，状态改为「需补充」，AI 复核附说明

### 收敛

所有 CL 条目均为「已解决」，且用户**明确告知代码审核通过**（如「审核 OK」「没问题」等）→ 报告状态改为「已审核」。

仅有 CL 全部解决但用户未明确确认 → 保持「待审核」，提示用户：**「待澄清项已全部解决，请确认代码审核结果」**。
